# V9 HERO10-CONTRACT1 英雄 XP / 升级分支合同

> 生成时间：2026-04-16
> 前置闭环：HERO9 死亡 + 复活分支（Task 212–218）已全部 accepted。
> 范围：Paladin XP、升级、技能点就绪合同。**不** 实现 XP 获取、升级运行时、技能点消费、能力升级、英雄 UI、AI 或任何生产代码。
> 本文档 **不** 声称"完整英雄系统"或"完整人族"。

---

## 1. 当前状态边界

### 1.1 Paladin 已有能力（HERO9 闭环后）

| 能力 | 来源 | 状态 |
|------|------|------|
| 圣骑士召唤 | HERO8/HERO9 | 运行时可用 |
| 圣光 (Holy Light) 施放 | HERO8 | 运行时可用 |
| 死亡状态 (`isDead=true`) | HERO9-IMPL1 | 运行时可用 |
| 祭坛复活 (Altar revive) | HERO9-IMPL2 | 运行时可用 |
| 复活费用/时间按等级缩放 | HERO9-IMPL2 | 运行时可用（使用 `heroLevel`） |

### 1.2 已存在但非完整 XP 系统的数据字段

以下字段存在于 `GameData.ts` 的 `UnitDef` 和 Paladin 定义中：

- `heroLevel?: number` — 英雄当前等级（Paladin 初始值 `1`）
- `heroXP?: number` — 英雄当前经验值（Paladin 初始值 `0`）
- `heroSkillPoints?: number` — 可用技能点（Paladin 初始值 `1`）

**合同声明**：这三个字段是 **数据种子**，不是完整 XP 系统。它们目前没有运行时逻辑驱动变化：
- 没有击杀奖励 XP 的运行时
- 没有升级判定逻辑
- 没有技能点消费路径
- `heroLevel` 仅被复活费用公式读取，永远不会从初始值 `1` 变化

### 1.3 当前运行时对 `heroLevel` 的使用

`Game.ts:5603` 中复活费用公式使用 `deadHero.heroLevel ?? heroDef.heroLevel ?? 1` 计算金币/时间缩放。这是 **HERO9 复活** 的已有行为，HERO10 XP 系统不得破坏此语义。

---

## 2. 合同条款

### 2.1 XP 获取必须作为未来运行时分支

XP 获取（击杀奖励、共享经验等）**必须** 在 `HERO10-SRC1` 中先明确来源边界，再在 `HERO10-IMPL1` 中实现运行时。本合同 **不** 定义：
- 具体 XP 表格
- 击杀奖励数值
- 等级阈值
- 技能点数值
- 能力等级效果

### 2.2 升级必须保留 HERO9 语义

XP 系统和升级运行时 **必须** 保留以下 HERO9 闭环语义不变：

| 语义 | 合同要求 |
|------|---------|
| 死亡状态 | 升级不改变 `isDead` 语义；死亡英雄不获得 XP |
| 复活费用按等级 | 升级后 `heroLevel` 变化应正确反映在复活费用公式中 |
| 唯一性 | 升级不创建新英雄记录；仍是同一个 Paladin 实例 |
| 无自动选择 | 升级不改变选择逻辑 |
| 圣光合法性 | 升级不改变圣光施放条件 |
| 无重复 Paladin | 升级不绕过唯一性约束 |

### 2.3 技能点和能力升级仅为就绪概念

本合同将 `heroSkillPoints` 定义为 **就绪数据字段**，但 **不** 实现：
- 圣光等级变化
- 光环 (Devotion Aura) 效果
- 终极技能 (Resurrection)
- 英雄 UI 升级面板
- 已学能力选择界面

### 2.4 本合同不改变生产代码

本合同仅产出文档和静态 proof。`Game.ts`、`GameData.ts`、运行时 spec、公共资源、board 代码 **不** 被修改。
