# M2-M7 User Return Packet：待人工判断总览

> 用途：用户离开一段时间后快速恢复上下文。本文只汇总 M2-M7 还需要哪些人工判断，不把任何里程碑标记为已人工通过。

## 1. 一句话总览

Codex/GLM 可以继续准备客观证据、修复可复现问题、整理文档和执行零行为变化硬化；但下面这些结论仍必须由用户确认：

- 核心规则感是否过关。
- 画面是否有 RTS 空间语法。
- 人机 Alpha 是否像一局可理解的对局。
- 项目下一阶段内容和视觉身份怎么选。
- 当前版本是否能私测 / 公开分享。
- 工程抽取是否开始影响玩家可见行为或产品方向。

## 2. M2-M7 摘要

| 里程碑 | 要判断什么 | 客观证据可以证明 | 仍需要用户判断 |
| --- | --- | --- | --- |
| `M2` 核心系统对齐 | 建造、箭塔、命令反馈、单位体积、战斗控制是否有 RTS 规则感 | build、typecheck、`test:m2`、相关 runtime packs、cleanup；核心合同没有明显破坏 | 这些系统组合起来是否符合直觉，是否可以进入视觉 / 空间判断 |
| `M3` 空间语法与可读性 | 当前画面是否开始像 War3-like RTS 战场，而不是网页原型 | scale、base grammar、camera/HUD regression；默认对象、镜头、反馈不崩 | 第一眼空间感、比例层级、镜头 framing、HUD 读图、proxy 视觉方向是否可接受 |
| `M4` 人机 Alpha 对局 | 能否打一局短 Alpha，并理解推进、胜负或卡点原因 | match-loop、AI recovery、AI economy、player-reported issues regression；核心操作和 AI 不崩 | 是否真的有开始、中期压力、可理解结束；控制是否公平；AI 是否低级坏死 |
| `M5` 内容与视觉身份方向 | 项目下一阶段应该优先长成什么样 | 方向 memo、scorecard、选项成本和风险可以准备 | 视觉方向、内容范围、下一个产品目标的最终选择；默认建议是否接受 |
| `M6` release / 私测判断 | 当前 build 是否可以外部试玩、少量私测或暂缓 | live smoke、known issues、build/typecheck、README/分享材料、red lines 可以准备 | 是否允许私测或公开分享；外部反馈是否会被已知问题误导 |
| `M7` 零行为变化硬化 | refactor 是否只降低工程风险，不改变玩家可见行为 | build、typecheck、runtime packs、diff 等价 review、cleanup、slice notes | 通常不需要产品判断；但一旦影响手感、视觉、AI、玩法或开发速度，需要用户介入 |

## 3. 推荐用户回看顺序

建议按这个顺序处理，不要先看 release：

1. `M2`：先确认核心规则感。规则不成立，后续画面和 release 判断都会被污染。
2. `M3`：再看第一眼空间语法、镜头和可读性。它决定 M4 playtest 是否能被正确理解。
3. `M4`：打一局短 Alpha，看是否有开始、中期压力、可理解结果或卡点。
4. `M5`：在 M2-M4 体感有基础后，决定默认方向是否采用：`hybrid + one polished Human slice + polish feel`。
5. `M6`：最后判断是否只做少量私测、公开分享，还是再等一个里程碑。
6. `M7`：只在工程抽取开始影响玩家可见行为、产品方向或开发速度时再人工介入；否则按工程证据 review。

如果时间很少，最短路径是：

1. 看 M3 第一眼。
2. 打一局 M4 短 Alpha。
3. 决定 M5 默认方向是否接受。
4. 决定 M6 是否允许少量私测。

## 4. 每个里程碑的用户结论格式

| 里程碑 | 建议只选一个结论 |
| --- | --- |
| `M2` | `通过` / `带债通过` / `失败：建造系统` / `失败：战斗系统` / `失败：命令卡反馈` / `失败：碰撞/体积` / `失败：战斗控制` |
| `M3` | `通过` / `通过但有美术债` / `失败：比例问题` / `失败：地形问题` / `失败：镜头问题` / `失败：视觉身份问题` |
| `M4` | `Alpha 通过` / `带债通过` / `失败：控制问题` / `失败：AI 问题` / `失败：节奏问题` / `失败：胜负不清晰` |
| `M5` | `hybrid + one polished Human slice + polish feel` / 或明确替换其中任一项 |
| `M6` | `公开分享` / `少量私测` / `再等一个里程碑` |
| `M7` | `接受该 extraction slice` / `拒绝` / `延期并转合同/测试任务` |

结论后最好补一句原因，例如：“M3 通过但有美术债：默认镜头可读，但 Farm 和 Barracks 比例还粗。”

## 5. 可以继续做什么

用户不在时，Codex/GLM 可以继续：

- 跑 build、typecheck、runtime packs、smoke path、cleanup。
- 修复可复现、可合同化的问题。
- 补 regression、整理已知问题、准备 review script 和 release 材料。
- 做零行为变化 extraction，并按 M7 review checklist 接受或退回。
- 记录候选建议、风险、默认路径和未决问题。

这些工作可以推进项目，但只能把里程碑状态推进到“客观证据已准备”。

## 6. 不能在用户回来前声称什么

没有用户明确结论前，不能声称：

- `M2` 已人工通过。
- `M3` 已确认有 War3-like 空间感。
- `M4` 已是一局可接受的人机 Alpha。
- `M5` 产品方向已正式决定。
- `M6` 可以公开分享，或私测已经被批准。
- `M7` 中任何影响玩家可见行为的 refactor 仍然只是 hardening。

如果某个问题能复现、能定义期望行为、能写 regression，就先回工程合同；如果问题需要判断“像不像、好不好、值不值得发”，就必须留给用户。
