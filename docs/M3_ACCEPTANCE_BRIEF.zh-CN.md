# M3 Acceptance Brief：空间语法、镜头与可读性

> 状态：`M3` 人工判断 brief。本文用于把 M3 gate packet 转成短验收口径，不代表 `M3` 已经人工通过。

## 1. M3 要锁定什么

`M3` 要锁定的是第一眼空间感：当前场景是否开始像一个 War3-like RTS 战场，而不是网页原型或调试沙盒。

具体要看三件事：

- 空间语法：基地、金矿、树线、出口、兵营的位置关系是否像有意设计。
- 镜头 framing：默认视角是否能同时读清基地核心、经济路径和主要操作区。
- 可读性：单位、建筑、选择圈、血条、HUD 和反馈是否支撑玩家理解场面。

M3 不要求最终美术完成，也不要求内容完整；它要求当前 proxy / fallback 视觉已经足够支撑后续 M4-M6 判断。

## 2. 客观可证明 vs 需要人工判断

| 范围 | 客观 / runtime 可证明 | 仍需人工判断 |
| --- | --- | --- |
| 基础稳定 | `npm run build`、`npx tsc --noEmit -p tsconfig.app.json`、相关 runtime packs 通过；本地浏览器 / Playwright / Vite 残留清理完成 | 打开后是否像一个 RTS 场景，而不是技术页面 |
| 尺寸与测量 | `m3-scale-measurement` 可验证单位、建筑、镜头或布局数据没有明显越界 | Worker / Footman / Farm / Barracks / Town Hall 的体量层级是否“看起来成立” |
| 基地语法 | `m3-base-grammar-regression` 可验证关键对象存在、相对位置和可达性不破坏合同 | 基地、金矿、树线、出口关系是否有 War3-like 空间意图 |
| 镜头与 HUD | `m3-camera-hud-regression` 可验证默认镜头、HUD、选择反馈不出现硬性遮挡或失效 | framing 是否舒服，HUD 是否破坏读图，第一眼是否能抓住主操作区 |
| 视觉方向 | runtime 可证明 proxy 元素加载、fallback 不崩、反馈可显示 | 这些 proxy 是否属于同一方向，是否值得继续朝 War3-like 推进 |

结论：客观包只能证明“可以让用户判断”；不能自动证明 M3 已通过。

## 3. 用户短评审应检查什么

建议用户只做一次短 M3 review，不需要完整打一局。重点看：

- 第一眼：是否马上读成 RTS 战场，而不是网页 demo 或随机摆物。
- 基地核心：Town Hall 是否是视觉中心，周围 worker、Farm、Barracks 是否有清楚层级。
- 经济路径：金矿和 Town Hall 的距离是否能读出采集路线，既不太远也不挤成一团。
- 树线和出口：边界、通道、可行动区域是否让地图有方向感。
- 镜头：默认视角是否覆盖主要操作区，是否需要一上来就拖镜头才能理解。
- HUD：是否遮挡战场关键对象，命令卡和状态信息是否帮助理解而不是抢画面。
- 反馈：选择圈、血条、移动/命令反馈是否清楚，是否和场景比例匹配。
- 视觉一致性：即便是 proxy，是否像同一个项目的临时视觉，而不是混搭调试资源。

## 4. M3 通过 / 带债通过 / 失败

| 结论 | 判定标准 | 后续处理 |
| --- | --- | --- |
| `通过` | 第一眼像 RTS 战场；基地 / 金矿 / 树线 / 出口有明确空间语法；比例、镜头、HUD、反馈足够支撑 M4 Alpha 对局判断 | 可以把 `M3` 记为人工通过，并把小问题转成后续视觉 polish 或 M5 方向讨论 |
| `通过但有美术债` | 空间语法和可读性成立，但 proxy 粗糙、细节不统一、部分比例或反馈还需要后续打磨 | 不阻止 M4；记录债务，避免在 M5/M6 中把这些 debt 误当最终视觉 |
| `失败：比例问题` | 单位和建筑体量层级明显不成立，影响读场和操作判断 | 回到 scale / visual contract，先修比例再评审 |
| `失败：地形问题` | 基地、金矿、树线、出口关系读不出来，像随机摆放或无方向空间 | 回到 base grammar / layout contract |
| `失败：镜头问题` | 默认 framing 看不到关键对象，HUD 挡住主战场，或一打开就无法理解操作区 | 回到 camera / HUD contract |
| `失败：视觉身份问题` | proxy 元素混乱到无法判断项目方向，或完全不像同一个 War3-like RTS | 先收敛视觉规则，再进入 M5 方向讨论 |

`通过但有美术债` 的前提是：用户仍能用当前画面判断空间、镜头和可读性。如果 debt 已经阻断判断，就应判失败。

## 5. 应回工程合同而不是争论视觉品味的问题

以下问题不要当成 taste 争论，应直接转成工程合同或 regression：

- 默认镜头看不到 Town Hall、金矿、worker 或主要操作区。
- HUD 遮挡关键对象、命令卡覆盖主战场，或在固定视口下不可读。
- 选择圈、血条、命令反馈不显示、显示错位或比例明显错误。
- 基地对象缺失、重叠、不可达，或金矿 / 树线 / 出口相对关系破坏操作路径。
- 视觉对象加载失败、fallback 缺失、运行时出现白屏、崩溃或持续错误。
- 同一视口、同一初始状态下布局或 framing 不稳定。

这些问题只要能复现、能定义期望行为、能写测试，就应回到 engineering contract。真正的 taste 讨论应留给“比例已经稳定但是否够好看 / 是否够 War3-like”的判断。

## 6. 用户回来前 Codex / GLM 可以继续什么

在用户返回前，Codex/GLM 可以继续准备 M3 的客观证据，但不能声称 `M3` 已人工通过：

- 维护 `m3-scale-measurement`、`m3-base-grammar-regression`、`m3-camera-hud-regression` 等客观包。
- 跑并记录 build、typecheck、相关 runtime packs 和本地残留清理结果。
- 把可复现的镜头、HUD、选择反馈、布局、加载问题拆成工程合同任务。
- 准备短 review 路径，让用户回来后快速判断第一眼空间感和可读性。
- 继续推进不冲突的 M4/M5/M6/M7 客观准备，但保持 `M3` 状态为“客观材料准备中 / 待用户人工裁决”。
- 不让 GLM 代替用户判断视觉品味，也不把测试通过等同于 M3 人工验收通过。
