# M7 Refactor Acceptance Brief：零行为变化硬化准入

> 状态：`M7` 工程硬化验收备忘。本文用于判断抽取/整理类工作是否可以作为 M7 hardening 接受，不用于批准产品方向、玩法调整或视觉品味改动。

## 1. 目的与范围

`M7` 的目的很窄：在不改变玩家可见行为的前提下，降低长期工程风险。

允许范围：

- 从 `Game.ts` 或其他膨胀边界中抽出可独立命名、可测试、可回退的子系统。
- 整理高风险视觉/反馈、选择、放置、清理、命令调度等逻辑边界。
- 补齐已有行为合同的验证入口，让后续改动更容易被测试拦住。

不允许范围：

- 借 refactor 调整玩法语义。
- 借 hardening 重做视觉品味。
- 借抽取改变 AI、资源、供给、建造、选择、HUD、路径或战斗的外部表现。

## 2. 什么算可接受的零行为变化抽取

一个 extraction slice 可以被接受，必须同时满足：

- 玩家操作结果不变：同样的输入、同样的初始状态，应得到同样的可见结果。
- 游戏合同不变：资源扣除、供给检查、采矿饱和、训练队列、建造中断/取消、AI 恢复、死亡清理等语义不发生漂移。
- 视觉/反馈意图不变：选择圈、命令反馈、HUD 展示、建造 preview、错误提示、单位出生和清理时机不能被顺手改掉。
- 抽取边界真实变清楚：新模块有单一责任，名称能解释它为何存在，而不是把旧代码复制到一个更难找的位置。
- 依赖方向更健康：主文件调用更薄，状态读写更集中，跨模块副作用没有变得更隐蔽。
- 回滚成本低：slice 小、文件范围可解释，出现行为差异时能单独回退。

可接受的例子：

- 把纯计算、查询、格式化、查找、命令分发 helper 抽到独立模块。
- 把已由 regression 覆盖的视觉/反馈创建逻辑移动到 factory 或 presenter。
- 把选择/放置/占地检查中已经存在的逻辑边界显式化，但不改变判定条件。
- 为已存在的合同补一个 focused regression，而不是扩大功能。

## 3. 应拒绝或移出 M7 的变化

下面这些变化不应作为 M7 hardening 合并：

- gameplay 语义变化：资源、供给、采集、训练、建造、攻击、AI、路径、胜负状态出现任何有意调整。
- 产品方向变化：把内容范围、阵营目标、公开 demo 包装或视觉身份混进 refactor。
- 视觉品味变化：颜色、比例、相机 framing、HUD 布局、单位/建筑可读性被顺手调整。
- 大范围重写：没有清晰 slice，多个子系统一起搬迁，review 无法判断行为是否等价。
- 未受保护的抽取：被移动逻辑没有对应 regression、smoke 或人工可复现路径。
- 新抽象过早：为了“看起来架构更好”新增复杂层，但没有降低真实风险。
- 隐性依赖增加：通过全局状态、事件、单例或循环 import 让行为更难追踪。
- 验证缩水：删除或弱化已有测试、跳过 runtime pack、只用 typecheck 证明安全。

如果改动看起来“顺便修了 bug”，默认不要作为 M7 接受；先把 bug 写成合同/测试任务，再决定是否修。

## 4. 接受 extraction slice 前必须有的证据

每个 slice 至少需要这些证据：

- 明确范围：改了哪些文件，抽出的边界是什么，哪些行为明确不变。
- 风险说明：涉及的子系统、可能受影响的合同、为什么认为是零行为变化。
- 对应验证：
  - `npm run build`
  - `npx tsc --noEmit -p tsconfig.app.json`
  - 相关 runtime regression pack
  - 若触及跨系统路径，再跑 `npm run test:runtime`
- 行为对照：reviewer 能从 diff 看出条件、时机、状态写入、对象生命周期没有改变。
- 本地卫生：验证后没有遗留 Vite / Playwright / Chromium 等 runtime 进程。
- 未决项记录：如果发现缺少合同覆盖，应记录为新的合同/测试任务，而不是靠人工相信抽取安全。

对于纯 docs 或 checklist 更新，不需要 runtime 验证；但只要 touched code，就不能只用文档理由接受。

## 5. Codex 如何 review GLM 的抽取工作

Codex review GLM 的 M7 slice 时，按这个顺序做：

1. 先看文件边界：是否只改了任务允许的模块，是否碰到 no-touch 文件，是否把多个子系统混成一单。
2. 再看行为等价：逐段对照条件分支、状态写入、集合迭代顺序、对象创建/释放时机、early return 和错误路径。
3. 检查副作用：是否新增隐藏全局状态、循环依赖、异步时序变化、事件重复注册或 cleanup 漏洞。
4. 检查测试证据：GLM 是否跑了约定命令，输出是否对应当前 slice，而不是泛泛说“tests pass”。
5. 检查命名和边界：新模块是否让下一次维护更容易，还是只是把 `Game.ts` 的复杂度搬走。
6. 决定结果：
   - 行为等价且证据足够：接受。
   - 行为可能变了但应该变：转成合同/测试任务，不能当 hardening 合并。
   - 行为可能变了且没有必要：退回或要求缩小。
   - 证据不足：要求补验证，或拆成更小 slice。

Codex 不应让 GLM 代替用户判断视觉品味或产品方向。只要 refactor 触及这些判断，M7 review 应停止在工程证据层，另开 gate 或任务。

## 6. 什么时候把 refactor 转回合同/测试任务

遇到以下情况，应把当前 refactor 改写为 contract/test task：

- 抽取过程中发现原行为不清楚，无法证明“原样搬迁”。
- 现有 regression 没覆盖关键路径，例如队列恢复、死亡清理、占地释放、AI 恢复、HUD 状态或资源边界。
- GLM 的改动暴露了真实 bug，但修复会改变玩家可见行为。
- review 需要依赖人工试玩才能判断是否等价。
- 两个实现都“可能对”，但项目缺少明确合同来决定哪个才是正确行为。
- refactor 后只能靠截图或主观感觉判断安全。

转换后的任务应先写清楚：

- 期望行为是什么。
- 当前行为是什么。
- 最小复现路径是什么。
- 应新增或更新哪类 regression。
- 修复是否仍属于 M7，还是应回到 earlier milestone 的行为合同。

## 7. 用户回来前可以继续什么

用户返回前，Codex/GLM 可以继续推进 M7 客观硬化，但不能把任何产品判断伪装成 refactor：

- 继续拆小、低风险、已覆盖的 extraction slice。
- 为高风险路径补 focused regression，再做抽取。
- 整理 `Game.ts` 风险边界和 review 记录，标出哪些地方还不能安全移动。
- 对 GLM 输出做严格等价 review，拒绝“顺便优化手感/视觉/AI”的硬化 PR。
- 保持 M7 的结论为工程可接受性判断；若发现需要用户感知判断，应暂停该 slice 并转入对应 gate。
