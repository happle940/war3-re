# 角色 02：执行制片 / 项目整合负责人

> 用途：定义“把方向变成推进节奏”的角色。  
> 当前状态：`已存在`  
> 当前 owner：Codex

## 1. 角色使命

这个角色负责把项目从“很多正确想法”变成“按版本推进的连续产出”。

它负责：

- 路线图拆解
- 版本节奏
- lane 拆分
- owner 分配
- 收口和合流

## 2. 阶段职责表

| 阶段 | 工作重点 | 主要交付 | 不负责 |
| --- | --- | --- | --- |
| `V0` | 建立持续执行模型 | queue、watch、handoff、验证顺序 | 不负责替代技术架构判断 |
| `V1` | 保证动词原型按依赖顺序落地 | 动词实现顺序、测试顺序、最小可玩链路 | 不负责每个动词的具体编码 |
| `V2` | 以“可信 slice”重新编排优先级 | contract 清单、gap 排序、收口顺序 | 不负责替代产品负责人做最终品味判断 |
| `V3` | 为战场切片拆出可并行泳道 | 战场语法、可读性、HUD、AI 的 lane 划分 | 不负责单点美术/UX 设计 |
| `V4` | 组织短局 alpha 验证节奏 | playtest 脚本、issue routing、gate packet | 不负责所有实测执行 |
| `V5` | 控制战略扩张范围 | tech/roster/upgrade 的波次推进计划 | 不负责具体平衡数值 |
| `V6` | 控制身份层引入风险 | hero / ability / neutral 的引入顺序 | 不负责具体能力设计 |
| `V7` | 转入 beta 式收口 | bug burn-down、冻结范围、验证矩阵 | 不负责每条 bug 的实现 |
| `V8` | 组织外部分享闭环 | smoke、README、known issues、反馈回流 | 不负责对外市场动作本身 |
| `V9` | 维护长期节奏 | patch cadence、债务清单、扩展路线 | 不负责最终产品方向定义 |

## 3. 当前阶段重点

当前重点是把项目从“只会做 M2-M7 切片”提升为：

1. 总版本路线清晰
2. 角色和泳道清晰
3. 每一轮执行都能映射到大版本推进

## 4. 交接与验收方式

- 输入：产品方向、仓库状态、worker closeout、测试结果
- 输出：队列、gate、任务包、合流顺序、风险登记
- 核心指标：持续推进，而不是局部忙碌
