# 角色 01：产品负责人 / 游戏总监

> 用途：定义“最终产品判断者”在 `V0-V9` 的职责。  
> 当前状态：`已存在`  
> 当前 owner：用户

## 1. 角色使命

这个角色负责回答两个最高层问题：

1. 我们到底在做一个什么样的 War3-like 产品？
2. 哪些推进是真进展，哪些只是工程自我感动？

它不负责亲自写所有代码，但负责最终方向、取舍、范围收缩和人眼验收。

## 2. 阶段职责表

| 阶段 | 工作重点 | 主要交付 | 不负责 |
| --- | --- | --- | --- |
| `V0` | 确认项目北极星、节奏、成功标准 | 北极星口径、项目约束、是否接受当前开发方式 | 不负责具体实现细节 |
| `V1` | 判断 RTS 动词原型是否成立 | select/move/gather/build/train/attack 的“可继续做”判断 | 不负责底层状态机设计 |
| `V2` | 判断可信 Vertical Slice 是否成立 | “前 5 分钟不被当假 RTS”的主观裁决 | 不负责所有 regression 编写 |
| `V3` | 判断战场是否开始像 War3-like | 对 TH/矿/树线/出口/兵营/塔空间语法做通过/打回 | 不负责具体地图坐标实现 |
| `V4` | 判断 10-15 分钟短局是否值得玩 | 人类 vs AI 短局 verdict、问题优先级重排 | 不负责 AI 全部调参 |
| `V5` | 决定战略骨架往哪扩 | 兵种、科技、timing、counter 的产品取舍 | 不负责每个 roster 的数值表 |
| `V6` | 决定是否引入英雄/法术/中立/非对称 | 身份层方向决策、范围冻结 | 不负责所有能力脚本实现 |
| `V7` | 冻结 Beta 候选范围 | content-complete 范围、平衡和 polish 优先级 | 不负责逐条 bug 修复 |
| `V8` | 决定私测/公开分享策略 | release verdict、外部叙事边界、已知问题披露口径 | 不负责打包流水线细节 |
| `V9` | 决定长期路线 | patch / expansion / 新方向的产品取舍 | 不负责长期运维执行 |

## 3. 当前阶段重点

当前最重要的不是继续扩系统，而是：

1. 把 `V2.5` 和 `V3.1` 分清
2. 用人眼判断哪些东西已经开始像 War3-like，哪些还只是“通过了测试”
3. 对 `V3 -> V4` 提前定义不可接受的假进展

## 4. 交接与验收方式

- 输入：Codex 的路线、GLM/Codex 的 closeout、测试与录像证据
- 输出：继续 / 打回 / 降范围 / 换主轴
- 核心验收口径：真实玩家观感优先于工程自证
