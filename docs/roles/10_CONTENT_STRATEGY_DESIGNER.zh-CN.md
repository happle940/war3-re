# 角色 10：内容 / 战略骨架设计师

> 用途：定义“当项目进入战略和内容扩张时，谁来决定做什么、不做什么”的角色。  
> 当前状态：`缺失`  
> 当前 owner：无稳定 owner

## 1. 角色使命

这个角色负责把项目从“最小可玩 RTS”推进到“有战略层的 RTS”：

- roster 扩张
- upgrade / prerequisite
- build order
- timing
- counter
- 内容范围冻结

## 2. 阶段职责表

| 阶段 | 工作重点 | 主要交付 | 不负责 |
| --- | --- | --- | --- |
| `V0` | 不主导，但需要参与约束未来数据结构 | 内容扩张边界提示 | 不负责基础设施实现 |
| `V1` | 不主导，但记录最小循环缺什么 | 初始 roster 缺口备忘 | 不负责原型动词实现 |
| `V2` | 不主导，但观察当前最小循环的战略缺失 | 最小战略缺口清单 | 不负责可信 slice 的收口 |
| `V3` | 开始介入空间与内容耦合判断 | 基地语法和兵种语义的关系建议 | 不负责地图/美术执行 |
| `V4` | 判断短局需要哪些最小战略选择 | 基础 build order、timing、对抗弧线建议 | 不负责 AI 具体实现 |
| `V5` | 成为主角 | 科技树、升级、兵种扩张、counter matrix、内容冻结范围 | 不负责所有 gameplay 编码 |
| `V6` | 决定身份层如何纳入战略 | hero/neutral/item/asymmetry 与战略骨架的关系 | 不负责能力脚本实现 |
| `V7` | 做 content-complete 收口 | 范围收缩、优先级、平衡方向 | 不负责所有数值微调 |
| `V8` | 决定对外版本展示哪些内容 | demo scope、禁用项、说明策略 | 不负责对外发布流程 |
| `V9` | 规划扩展方向 | 新阵营、新地图池、新模式的内容路线 | 不负责整体产品路线 |

## 3. 当前阶段重点

这个角色现在还不是主角，但必须尽早定义，否则后面会乱长。

当前应准备：

1. `V5` 之前的战略缺口账本
2. 未来 roster / tech / timing 的扩张原则
3. 什么叫“像 War3-like 的战略层”，什么只是堆内容

## 4. 交接与验收方式

- 输入：当前系统、比赛录像、体验缺口、War3-like 参考标准
- 输出：内容范围、战略骨架、冻结策略、counter 框架
- 核心指标：内容增长带来策略深度，而不是复杂度噪音
