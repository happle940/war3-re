# 角色 06：HUD / UX / 信息架构设计师

> 用途：定义“让玩家看得懂、点得准、理解系统反馈”的角色。  
> 当前状态：`缺失`  
> 当前 owner：无稳定 owner

## 1. 角色使命

这个角色负责信息是否被正确传达：

- 选中态
- 命令卡
- disabled reason
- 建造/训练状态
- 战斗反馈
- HUD 和战场的遮挡关系

## 2. 阶段职责表

| 阶段 | 工作重点 | 主要交付 | 不负责 |
| --- | --- | --- | --- |
| `V0` | 确立基本 HUD 骨架和调试信息边界 | HUD 分区、调试层边界、最小信息层级 | 不负责底层 engine 实现 |
| `V1` | 支撑动词可发现性 | 选择反馈、命令按钮、右键语义、状态显示 | 不负责全部命令逻辑 |
| `V2` | 让命令可信可解释 | HUD/selection sync、disabled reason、资源/补给提示 | 不负责产品最终决策 |
| `V3` | 让 HUD 与战场空间协调 | 相机、面板、点击区、遮挡、建造 footprint 提示 | 不负责地图空间设计 |
| `V4` | 支撑完整短局信息节奏 | 压力状态、告警、生产/补兵/战斗反馈 | 不负责 AI 行为设计 |
| `V5` | 承载更多战略系统 | 科技、升级、队列、编队、更多单位信息 | 不负责科技树内容本身 |
| `V6` | 承载身份层复杂度 | 英雄面板、法术、mana、物品、中立提示 | 不负责能力数值 |
| `V7` | 做 usability polish | 新玩家理解路径、提示文案、反馈一致性 | 不负责所有外部教程 |
| `V8` | 面向外部试玩收口 | README/HUD 认知一致、已知 UX 问题披露 | 不负责发布流程 |
| `V9` | 长期优化信息架构 | 新内容纳入 HUD 体系、历史债务清理 | 不负责总体 roadmap |

## 3. 当前阶段重点

当前最该补的不是“更酷的 HUD”，而是：

1. 当前信息是否足以支持正确操作
2. 玩家失败时能否知道原因
3. HUD 是否在破坏 `V3` 的战场第一眼判断

## 4. 交接与验收方式

- 输入：录像、失败操作案例、命令卡状态、主观体验
- 输出：信息层级、按钮/文案语义、反馈规范、交互修正
- 核心指标：玩家知道自己现在能做什么、为什么不能做
