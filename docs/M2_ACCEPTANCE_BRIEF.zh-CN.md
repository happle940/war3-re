# M2 Acceptance Brief：核心系统规则感验收

> 状态：`M2` 人工判断 brief。本文用于把 M2 gate packet 转成短验收口径，不代表 `M2` 已经人工通过。

## 1. M2 要锁定什么

`M2` 要锁定的是核心 RTS 系统是否已经对齐到“玩家操作有规则感”的程度。

重点不是画面是否好看、AI 是否聪明、对局是否完整，而是这些基础系统是否符合 Warcraft-like RTS 玩家预期：

- 建造生命周期：中断、恢复、取消、退款、占地释放、HUD 清理。
- 防御建筑：箭塔能索敌、攻击、换目标，但不追击、不乱打友军或建筑。
- 命令反馈：人口、金币、木材不足时，命令卡能说清为什么不能点。
- 单位存在感：多个单位不会坍缩成一个点，移动和采集保持基本体积感。
- 战斗控制：移动、停止、驻守、attack-move 与 auto-aggro 不互相抢控制权。

如果这些系统不稳定，后续视觉、Alpha 对局、内容方向和 release 判断都会被污染。

## 2. 客观可证明 vs 需要人工判断

| 范围 | 客观 / runtime 可证明 | 仍需人工判断 |
| --- | --- | --- |
| 自动化入口 | `npm run build`、`npx tsc --noEmit -p tsconfig.app.json`、`npm run test:m2`、cleanup 全部通过 | 这些绿灯是否足够支撑一次人工确认 |
| 建造生命周期 | regression 可证明中断恢复、取消退款、占地释放、HUD 清理 | 玩家是否觉得建造恢复和取消符合直觉 |
| 静态防御 | regression 可证明箭塔索敌、伤害、目标切换、不追击、不攻击友军 / 建筑 | 箭塔是否像防御建筑，而不是摆设或乱跑单位 |
| 命令卡反馈 | regression 可证明人口、金币、木材不足时按钮状态和文案刷新 | 用户是否能不用猜就理解“为什么点不了” |
| 单位存在感 | regression 可证明单位分离、间距和采集防坍缩 | 画面上是否有足够实体感，不像多个单位叠成一个 |
| 战斗控制 | regression 可证明移动覆盖攻击、stop 清链、hold 不追击、suppression 窗口成立 | 打架中的单位是否仍然“听话”，玩家是否感觉控制权在自己手里 |

结论：M2 自动化可以证明核心合同没有明显破坏；人工确认要判断这些合同组合起来是否已经有 RTS 规则感。

## 3. 用户短 M2 review 应检查什么

建议按 10-15 分钟脚本快速过一遍，不需要评价画面、AI 强度或完整胜负闭环。

重点看：

- 建造恢复：农民停工后，另一个农民是否能自然接手。
- 建造取消：未完成建筑取消后，建筑、资源、占地和农民状态是否符合直觉。
- 箭塔行为：箭塔是否像固定防御建筑，能打敌人但不会追出去。
- 命令卡反馈：按钮不可用时，是否能看懂是人口、金币还是木材问题。
- 单位体积：4-5 个单位走到同一点后，是否仍有可接受间距。
- 战斗控制：战斗中右键移动、按 S 停止、按 H 驻守时，单位是否执行玩家指令。
- 自由试玩：组合操作几分钟后，是否出现明显违背 RTS 直觉的系统行为。

用户只需要判断“规则是否基本对齐”，不要在 M2 阶段扩大到视觉方向或完整对局体验。

## 4. M2 通过 / 带债通过 / 失败

| 结论 | 判定标准 | 后续处理 |
| --- | --- | --- |
| `通过` | 五类核心系统都符合基本 RTS 预期；玩家能理解操作结果；没有明显系统规则违和 | 可以把 `M2` 记为人工通过，并继续后续里程碑 |
| `带债通过` | 核心规则成立，但视觉/手感/文案粗糙，或个别体验债不阻断系统判断 | 记录债务；不阻止进入后续工作；不要把债务误写成 M2 合同失败 |
| `失败：建造系统` | 建造中断、恢复、取消、退款、占地释放或 builder 状态存在可感知错误 | 回工程合同，补复现和 regression |
| `失败：战斗系统` | 箭塔索敌、目标切换、伤害、不追击或友军过滤不符合防御建筑规则 | 回静态防御 / 战斗合同 |
| `失败：命令卡反馈` | 不可用命令原因不清楚、不准确、不刷新，导致用户靠猜 | 回 command-card state contract |
| `失败：碰撞/体积` | 单位明显堆叠、挤压、采集坍缩，破坏基本实体感 | 回 unit presence / spacing contract |
| `失败：战斗控制` | 战斗中单位不听移动、停止、驻守，或 auto-aggro 太快抢回控制 | 回 combat-control contract |

`带债通过` 只适用于“规则已经成立但外观或手感粗糙”。如果规则本身错了，应判失败。

## 5. 应回工程合同而不是产品 taste 的失败

以下问题不要争论“喜不喜欢”，应直接转成 engineering contract / test task：

- 建造取消没有退款、占地没释放、农民状态卡住、未完成建筑 HUD 残留。
- 中断后无法续建，或 builder 切换导致建筑生命周期异常。
- 箭塔攻击友军、攻击建筑、追出防御范围、敌人死后不换目标或持续打空引用。
- 命令按钮可用性与资源 / 人口状态不一致，或原因文案错误。
- 训练、建造、采集、移动等同一操作在相同状态下结果不稳定。
- 单位重叠严重、采集点坍缩、群体间距不确定，影响点击和读图。
- 移动命令不能覆盖攻击，stop 不清动作链，hold 会追击，auto-aggro 抢控制。

只要能复现、能定义期望行为、能写 regression，就应回工程合同处理。M2 的 taste 判断只保留给“系统都对了，但组合起来是否像 RTS”的人工确认。

## 6. 用户回来前 Codex / GLM 可以继续什么

在用户返回前，Codex/GLM 可以继续准备 M2 的客观证据，但不能声称 `M2` 已人工通过：

- 维护 `npm run test:m2` 覆盖的五组核心 spec，并保持 32 个断言可重复通过。
- 继续跑并记录 build、typecheck、M2 runtime packs 和本地残留清理结果。
- 把可复现的建造、箭塔、命令卡、单位体积、战斗控制问题拆成合同任务。
- 保持相邻支撑合同稳定，例如 mining saturation、rally、command surface、AI economy。
- 准备短 review 脚本，让用户回来后能快速做规则感判断。
- 继续推进不冲突的后续客观工作，但保持 `M2` 状态为“自动化入口已准备 / 待用户人工裁决”。
- 不让 GLM 代替用户判断“规则感是否已经过关”，也不把自动化全绿等同于人工验收通过。
