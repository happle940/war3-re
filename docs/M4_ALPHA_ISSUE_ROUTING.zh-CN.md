# M4 Alpha Issue Routing：问题分流规则

## 目的

M4 alpha 问题先按证据分流，不直接把“感觉不好”当作口味争论。只要问题能落到输入、命令、经济、AI、建造、HUD、胜负或清理规则，就应回到合同/测试和实现修复；只有在客观规则已经成立后，才保留给用户做人类 alpha 判断。

## 分流矩阵

| 问题类别 | 默认路由目标 | 改路由前必须有的证据 | 说明 |
| --- | --- | --- | --- |
| 控制、选中、右键、命令队列 | 先转合同/测试任务；测试固定后再交 GLM 实现 | 具体操作顺序、起始选择状态、期望命令、实际行为、是否可复现、最接近的现有命令回归面 | 例如单位不响应、右键丢失、stop/hold/attack-move 语义错误、shift 队列被吞。 |
| 资源、供给、训练、采集 | 先转合同/测试任务；实现修复可交 GLM | 资源/供给前后数值、按钮或命令、单位/建筑类型、期望扣费和产出、运行时复现路径 | 例如资源未扣、供给封顶仍能训练、训练完成不出生、农民采集循环卡住。 |
| 建造、中断、取消、占位 | 先转合同/测试任务；实现修复可交 GLM | 建造者状态、建筑进度、取消/恢复动作、退款/占位/选择/HUD 状态 | 例如右键无法恢复建造、取消未释放 footprint、未完成建筑能攻击、建造者被卡死。 |
| AI 活动与恢复 | 先转合同/测试任务；实现修复可交 GLM | 时间线、AI 工人/资源/供给/波次状态、受到打击前后状态、期望的 bounded recovery | 例如 AI 不采集、不训练、首波或二波停摆、损失工人后终局崩溃、重复无效造房。 |
| 胜负与卡局 | 可复现状态错误转合同/测试；表达清晰度留给用户判断 | 触发条件、玩家/AI 存活状态、HUD 或 overlay 文案、是否仍能操作、复现步骤 | 胜负状态错误、游戏无法结束、明明结束仍可继续操作，应工程处理。 |
| HUD、反馈、可读性 | 错误/缺失/过期状态转合同/测试；“够不够清楚”留给用户 | 截图或短视频、当前选择对象、按钮状态、提示文本、对应系统状态 | 例如按钮可点但命令必失败、资源提示与真实数值不一致、死亡对象残留血条。 |
| 压力、节奏、公平感 | 默认保留给用户判断；若能定位到客观系统缺陷则回合同/测试 | 一局短测时间线、用户被阻塞的位置、是否能指向 AI/控制/HUD/经济具体规则 | “太快/太慢/太强/太弱”本身不是合同 bug；背后若是 AI 不恢复或控制失灵，则改路由。 |
| 视觉风格、内容量、打磨程度 | 延后或用户判断，不作为 M4 工程修复入口 | 说明它是否阻碍 alpha match-loop 判断；若只是美术或内容期待，标为后续方向债 | M4 只判断 match-loop readiness，不用在这里决定 M5 内容方向或 M6 发布包装。 |
| 多系统交叉、归因不清 | 留给 Codex 集成分诊 | 最小复现、影响系统列表、是否已有测试覆盖、是否需要拆成多个合同 | Codex 先拆成可验证切片，再决定交 GLM、补合同，或退回用户判断。 |

## 改路由前的证据要求

每个问题至少记录：

- 症状：用户看到或感到的问题是什么。
- 操作：从空闲状态到问题发生的最短步骤。
- 期望：按当前 M4 合同应发生什么，而不是“我希望更好玩”。
- 实际：实际发生的状态、数值、HUD、单位行为或 AI 行为。
- 复现性：一次偶发、稳定复现，还是只在某个局面复现。
- 证据：截图、短视频、控制台错误、测试名、或可写入测试的状态断言。
- 归属：合同/测试、GLM 实现、Codex 集成，或用户判断。

没有这些证据时，不应直接把问题派给 GLM 改实现；先由 Codex 收敛复现或把它降级为用户判断记录。

## 看起来主观、但实际是合同 bug 的例子

- “单位手感很不听话”：如果 move/stop/hold/attack-move 被覆盖、丢队列或错误追击，这是命令合同问题。
- “AI 像死了一样”：如果 AI 不采集、不训练、不出波、不恢复工人损失，这是 AI 活动或恢复合同问题。
- “我不知道为什么不能造兵”：如果按钮状态、资源/供给提示或失败反馈错误，这是 HUD/命令面合同问题。
- “战斗很不公平”：如果根因是选中丢失、命令无效、塔攻击状态错误、AI 波次异常，这是工程问题。
- “游戏卡住了”：如果胜负无法结算、死亡对象残留、占位未释放、资源目标失效，这是回归合同问题。
- “建造系统很怪”：如果中断后不能恢复、取消退款错误、未完成建筑产生战斗能力，这是建造生命周期问题。
- “反馈看不懂”：如果血条、选择圈、命令卡、状态文本与真实模拟状态不一致，这是可测试的反馈问题。

## 应保留给用户判断的例子

- 在客观命令和 AI 合同成立后，当前压力是否足够像一局 alpha。
- AI 强度是否让短局有趣，而不是只是在数值上可运行。
- 胜负已经正确显示后，结束反馈是否足够有满足感。
- 单局节奏是否应该更快、更慢，或更偏向防守/进攻。
- 代理视觉和反馈都没有状态错误时，当前可读性是否足够接受。
- M4 是否可以“带债通过”，以及哪些债务必须进入后续里程碑。

## 路由目标规则

### 回到合同/测试任务

满足任一条件就应回合同/测试：

- 行为可复现，且能写出期望状态断言。
- 问题影响控制权、资源守恒、供给上限、建造生命周期、AI 恢复、死亡清理或胜负状态。
- 问题已经被用户描述成体验问题，但根因能落到可验证规则。

### 回到 GLM 实现

只有在合同边界清楚后才交 GLM：

- 已有测试失败，或 Codex 已给出可执行的最小修复合同。
- 修复范围局限在明确实现区域。
- 不要求 GLM 改 M4 判定标准、重排里程碑，或替用户做 alpha 口味判断。

### 留给 Codex 集成

以下情况先由 Codex 接住：

- 一个问题横跨控制、HUD、AI 或胜负多个系统。
- 现有测试名不够贴合，需要先拆出新合同。
- GLM closeout 证据不足，需要复核、补测或重新分派。
- 修复看似小，但可能改变 M4 行为边界。

### 延后给用户判断

以下问题不应被工程化争论：

- 没有客观错误，只是在当前 alpha 质量上做 pass/pass with debt/fail 判断。
- 内容量、视觉身份、长期平衡、公开展示适配等不属于 M4 match-loop readiness 的问题。
- 当前系统正确运行，但用户需要决定“是否已经值得进入下一阶段”。

## 记录格式

建议每个 M4 alpha 问题用下面格式记录：

```text
问题：
证据：
复现步骤：
期望：
实际：
初始分类：
路由目标：合同/测试 | GLM 实现 | Codex 集成 | 用户判断
阻塞级别：阻塞 M4 | 可带债通过 | 后续债务
下一步：
```

没有明确证据的问题可以先记录，但不能据此宣称 M4 human-approved，也不能把用户未确认的体验判断写成已通过结论。
