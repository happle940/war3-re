# War3 RE - 给 GLM5.1 的执行接力文档

## 1. 这份文档的目的

这不是新的宏大路线图，而是 `ROADMAP_STRATEGY.md` 和当前代码现实之间的桥接文档。

`ROADMAP_STRATEGY.md` 的方向总体是对的：

- 北极星目标清楚
- 版本节奏清楚
- 范围控制意识正确
- 知道 AI、寻路、资产、手感才是关键

但当前代码还停留在“能跑的原型”，没有进入“可持续迭代的 RTS 基座”阶段。  
所以你接下来不能继续按“加功能”的思路推进，而要先做一次 **基线稳定化 + 运行时真相统一 + 执行边界收紧**。

一句话判断：

> 这个项目当前最缺的不是更多功能，而是一个能承载后续功能的稳定骨架。

## 2. 基于真实代码确认的项目现状

以下结论来自当前仓库代码，不是推测：

### 2.1 当前不是“Phase 0 后期”，而是“Pre-Phase-0 原型”

虽然已经有：

- W3X 地图解析
- 基础地形渲染
- 单位/建筑/采集/建造/训练/战斗/HUD

但这些能力还没有形成真正的系统边界。当前核心逻辑仍然集中在一个文件里：

- `src/game/Game.ts`：1503 行

这个文件同时负责：

- 游戏循环
- 输入
- 单位状态机
- 战斗
- 建造
- 训练
- 资源
- 选择
- HUD
- 小地图
- 地图加载
- 场景对象管理

这说明项目现在的真实阶段是：

> “一个高度可玩的原型”  
> 而不是  
> “已经完成架构打底、可以稳定扩展的 RTS 项目”

### 2.2 当前主干甚至还不稳定

`npm run build` 目前失败。

已确认的编译错误：

- `src/game/Game.ts:1354` 读取了 `CameraController.viewSize`
- `src/game/CameraController.ts` 并没有这个属性

这件事很重要，因为它说明目前连“可构建基线”都没有锁住。  
后续任何大重构如果建立在一个不可稳定构建的主干上，返工概率会非常高。

### 2.3 现在最大的架构问题不是“大文件”，而是“运行时真相分裂”

项目当前至少有下面几组“双重真相”：

#### A. 地图真相分裂

- `this.terrain` 仍然持有旧程序化地形
- `this.w3xTerrain` 持有 W3X 高度数据
- 地图加载后，部分功能基于 W3X，部分功能仍然基于旧 `Terrain`

直接后果：

- `getWorldHeight()` 会走 `w3xTerrain`
- 但 `updateTileInfo()` 仍然读 `this.terrain`
- `updateMinimap()` 仍然遍历 `this.terrain.width/height` 和 `this.terrain.getTile()`
- 地面射线检测虽然被替换成了新的 `groundPlane`，但地图语义并没有统一切换

也就是说：

> 现在“地图加载成功”不等于“游戏已经真正运行在该地图上”

#### B. 模拟真相分裂

当前单位的权威状态直接挂在渲染对象上：

- 位置来自 `unit.mesh.position`
- 旋转来自 `unit.mesh.rotation`
- 高度来自 Three.js 场景查询逻辑

这会让后续这些系统非常难干净接入：

- 寻路
- 碰撞
- 命令队列
- AI
- 回放
- 断言测试
- 固定步长模拟

原因很简单：

> 现在“游戏状态”不是独立数据，而是 Three.js 场景对象本身

#### C. 阵营真相分裂

解析层已经能拿到多个玩家出生点，但玩法层仍然几乎是“team 0 专属逻辑”：

- HUD 资源只看玩家 team 0
- 选择框只选 team 0
- 训练 UI 只读 team 0
- 建造找农民只找 team 0
- 右键逻辑把 `target.team !== 0` 都当敌人
- `TEAM_COLORS` 只有两个颜色

这意味着：

> 当前代码真实只支持“1个玩家 + 粗糙敌方目标”，并不支持真正的多玩家地图语义

### 2.4 当前 W3X 支持是“可展示”，不是“可驱动玩法”

`src/map/W3XParser.ts` 和 `src/game/Game.ts` 当前已经做到：

- 读取 `w3e`
- 读取 `w3i`
- 粗略读取 `doo`
- 根据出生点生成初始建筑/农民

但还没做到：

- 用 `doo` 中真实单位/装饰物做权威落点
- 用 `wpm` 或统一 blocker 数据驱动通行性
- 用地图对象真实分布驱动资源/树木/障碍
- 让 minimap / tile info / path grid / build validation 都以同一个 map runtime 为准

所以现阶段最准确的描述是：

> W3X 已经接入“展示链路”，但还没有接入“玩法链路”

### 2.5 当前还有明确的生命周期和内存风险

已经确认的风险点：

- `loadMap()` 新建 `W3XTerrainRenderer`，但没有持久保存和统一回收旧 renderer
- 再次加载地图时，旧的 `renderer.group` 很可能残留在场景中
- 单位死亡、地图切换时移除了 mesh，但没有系统性 dispose 子 mesh / geometry / material
- 血条移除时也只是 `scene.remove()`，没有完整释放 GPU 资源

这类问题在前期不一定立刻爆炸，但会让你很难判断：

- 是逻辑 bug
- 还是重复对象残留
- 还是 GPU 内存泄漏

### 2.6 当前还没有“命令层”

现在玩家输入会直接改单位运行状态：

- 右键地面，直接写 `moveTarget`
- 右键金矿，直接写 `gatherType/state`
- 右键敌人，直接写 `attackTarget/state`
- 建造直接写 `buildTarget/state`

这会导致一个非常关键的问题：

> 玩家、AI、脚本、快捷键，未来都不能通过同一套命令入口驱动游戏

而 RTS 的真正稳定骨架通常是：

- 输入层发命令
- AI 层发命令
- 命令系统消费命令
- 单位系统执行命令并改变状态

如果这一层没有建立，后面无论做 AI 还是编队，都会越来越乱。

## 3. 对 `ROADMAP_STRATEGY.md` 的补充判断

这份路线图总体方向没有问题，但还缺三件非常关键的东西：

### 3.1 缺一个真正的 Cycle 0

路线图从 “ECS + EventBus + MovementSystem” 开始想是合理的，但按当前代码状态，正式开始之前必须加一个：

> Cycle 0：稳定基线 / 真相统一 / 可验证化

否则你会在“架构重构”和“现有原型 bug/残留状态”同时作战，复杂度不必要地上升。

### 3.2 路线图低估了“命令层”的优先级

路线图正确强调了：

- ECS
- EventBus
- 寻路
- TeamResources

但当前代码最缺的还有一层：

> Command Model（Move / Attack / Gather / Build / Train）

没有这层，AI、输入、编队、未来热键系统都只能继续直接写单位内部状态，架构会继续失真。

### 3.3 路线图还没明确“模拟状态不能依赖 Three.js 对象”

如果继续在 `mesh.position` 上堆逻辑，会出现一种很危险的假重构：

- 文件拆了
- 类变多了
- 看起来像 ECS 了
- 但本质仍是渲染对象驱动模拟

这会让后面大部分系统依旧耦合。

所以必须补一句总原则：

> Three.js 负责表现，游戏数据负责真相。  
> 即便短期内不能完全分离，也必须按这个方向推进。

## 4. 现在不要做什么

以下事情现在都不应该优先：

- 不要先做英雄
- 不要先做更多单位
- 不要先做战争迷雾
- 不要先做音效
- 不要先做人族 T2/T3
- 不要先做资产管线
- 不要一上来重写完整 ECS 全家桶
- 不要一口气拆出十几个系统文件

原因不是这些不重要，而是：

> 当前骨架还没有稳到足以支撑这些内容继续叠加

如果在这个阶段继续加内容，只会把当前原型从“一个大文件”升级成“多个边界不清的小文件”。

## 5. 新的执行顺序

把后续执行顺序改成下面这样：

### Cycle 0：稳定基线

目标：

- 主干恢复为可构建
- 当前地图/单位/场景的运行时真相统一
- 地图多次加载不残留对象
- 现有原型仍然能玩

这一轮不追求“架构漂亮”，只追求：

- build 通过
- 地图链路可信
- 场景生命周期清楚
- 当前系统有可验证基线

### Cycle 1：抽出最小运行时骨架

目标：

- 从 `Game.ts` 中先抽出最少但最关键的权威数据结构
- 不要求一步到完整 ECS
- 优先抽：
  - `TeamResources`
  - `GamePhase`
  - `CurrentMapRuntime`
  - `UnitRuntime` / `BuildingRuntime` 的最小公共状态

重点不是“组件化形式”，而是：

> 先把权威数据从 Three.js 对象中松开

### Cycle 2：建立命令层

目标：

- 输入不再直接改单位内部状态
- 所有操作先变成命令

至少先统一五类命令：

- `move`
- `attack`
- `gather`
- `build`
- `train`

这样做完以后：

- 玩家输入和 AI 才能走同一入口
- 单位状态机才不会继续在多个入口处被直接篡改

### Cycle 3：导航与占用

目标：

- 建立 blocker / occupancy grid
- 建筑放置校验
- 基础 A*
- 基础碰撞分离

注意这里的顺序是故意的：

> 先有统一地图语义和命令，再做寻路  
> 会比直接把 A* 塞进当前原型里稳很多

### Cycle 4：AI 雏形

只有当以下三件事都成立后，AI 才值得开始：

- 命令层已存在
- 地图 blocker 语义已存在
- per-team 资源和基础生命周期已存在

否则 AI 只会变成一堆对当前 `Game.ts` 私有细节的脆弱直连代码。

## 6. 你下一步的唯一任务：先做 Cycle 0

这是你现在应该立刻执行的任务。  
本轮只允许做“稳定基线”，不要顺手扩功能。

### 6.1 本轮目标

把当前项目从“能跑但不稳的原型”提升为“可继续重构的稳定基线”。

### 6.2 本轮必须完成的事项

#### 任务 A：修复构建失败

要求：

- 修复 `npm run build` 的 TypeScript 错误
- 明确小地图视口框应如何从 `CameraController` 获取
- 不要用临时硬编码糊过去

#### 任务 B：统一当前地图运行时入口

要求：

- 明确“当前地图”的单一真相
- 加载 W3X 后，以下功能必须都基于当前地图，而不是旧 `Terrain`
  - 地面拾取
  - tile info
  - minimap
  - 地图尺寸
  - 高度查询

你可以保留旧 `Terrain` 作为 fallback，但不能让它和 W3X 同时充当权威来源。

#### 任务 C：修正地图切换生命周期

要求：

- 多次 `loadMap()` 不得叠加旧 terrain renderer
- 场景残留对象要有清理入口
- 旧几何体、材质、纹理应被明确释放

#### 任务 D：补一个最小 smoke baseline

要求：

- 记录并实现一份最小验证清单
- 这份清单至少覆盖：
  - 默认启动可进入场景
  - 自动加载测试地图成功
  - 单位可选中、可右键移动
  - 农民可采集
  - 建筑可训练单位
  - `npm run build` 成功

这份清单可以先写进文档，不要求完整自动化测试。

### 6.3 本轮禁止事项

- 不要引入完整 ECS 框架
- 不要开始 AI
- 不要新增单位/建筑/英雄
- 不要做资产加载
- 不要做战争迷雾
- 不要顺手改 UI 风格
- 不要为了“以后方便”一次性建很多空目录和空抽象

### 6.4 本轮交付约束

- 修改文件尽量控制在 4~6 个
- 总新增代码尽量控制在 1200 行以内
- 每个新增文件都必须有明确职责
- 每个 Three.js 新资源都要想清楚 cleanup

### 6.5 本轮完成定义（DoD）

只有同时满足下面全部条件，Cycle 0 才算完成：

- `npm run build` 通过
- 连续加载两次不同地图，场景不叠层、不明显残留
- minimap / tile info / 地图尺寸与当前地图一致
- 默认玩法链路仍然可玩
- 没有再新增一个新的“大一统上帝文件”

## 7. 建议的最小实现策略

为了避免你在 Cycle 0 又写成“大重构”，建议这样做：

### 7.1 先做薄抽象，不做大迁移

本轮优先做薄层封装，例如：

- `CurrentMapRuntime`
- `disposeObject3DDeep()`
- `CameraController` 暴露稳定视口查询接口

而不是：

- 一口气做完整 World / Entity / Component / System

### 7.2 先收口接口，再迁移逻辑

建议顺序：

1. 修 build
2. 收口“当前地图查询接口”
3. 收口“场景对象回收接口”
4. 把 minimap / tile info / loadMap 这些消费者切到新接口

这会比直接改所有系统安全得多。

### 7.3 先保证正确，再讨论优雅

当前阶段最值钱的不是“最漂亮的架构”，而是：

- 代码能 build
- 地图加载语义不混乱
- 重构不会因为旧残留对象而误判

## 8. 给你的代码风格约束

后续每个 Cycle 都遵守：

### 8.1 一次只解决一个主问题

每个 Cycle 只允许有一个主目标，例如：

- “稳定地图 runtime”
- “抽出 team resources”
- “建立命令层”

不要在同一轮里同时做：

- 路径系统
- AI
- UI 重写
- 资产替换

### 8.2 不允许假抽象

以下情况都算假抽象：

- 文件拆了，但依赖关系更乱
- 新建了 System 类，但内部仍直接操作 `mesh.position` 作为唯一真相
- 加了 EventBus，但大部分状态仍直接互相改引用
- 新增目录很多，但只是把原逻辑平移过去

### 8.3 每轮都必须给出“这轮故意没做什么”

交付时必须明确写出：

- 本轮完成了什么
- 本轮没做什么
- 为什么现在不做

这样可以抑制范围蔓延。

## 11. 最后的执行原则

你后面每次开发都要记住：

> 先把“可继续开发”做出来，再把“更像 war3”做出来。  
> 先稳住基线，再堆玩法。  
> 先统一真相，再谈架构优雅。

如果你照这个顺序推进，`ROADMAP_STRATEGY.md` 仍然可以作为主路线图。  
但在真正进入它的 Cycle 1 之前，必须先完成这里定义的 Cycle 0。
