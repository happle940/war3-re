# M4 Alpha Acceptance Brief：人机 Alpha 对局验收

> 状态：`M4` 人工判断 brief。本文用于把 M4 gate packet 转成更清晰的 alpha match-loop 验收口径，不代表 `M4` 已经人工通过。

## 1. M4 要证明什么

`M4` 要证明的不是“游戏已经完整”，而是：

**玩家可以打一局短的人机 Alpha，并理解这局为什么推进、为什么赢、为什么输、为什么卡住。**

M4 的核心是 match loop readiness：开局成立，中期有压力，结局或卡点能被理解；玩家没有因为控制、AI 崩坏或胜负表达缺失而失去判断基础。

## 2. 客观可证明 vs 需要人工判断

| 范围 | 客观 / runtime 可证明 | 仍需人工判断 |
| --- | --- | --- |
| 启动与基础稳定 | `npm run build`、`npx tsc --noEmit -p tsconfig.app.json`、相关 runtime packs 通过；本地浏览器 / Playwright / Vite 残留清理完成 | 打开后是否像一个可玩的 Alpha，而不是技术样机 |
| 对局结构 | regression 可验证 AI 会开局、恢复、采集、出兵，玩家核心操作不崩 | 是否真的有“开始阶段 / 中期压力 / 明确结束或明确卡点” |
| 控制与公平 | 测试可覆盖选择、命令、队列、建造、资源、供给等合同 | 控制问题是否让玩家觉得不公平、无法防守、无法补兵或无法反打 |
| AI 完整性 | `m4-ai-recovery`、`ai-economy` 等可证明 AI 不因基础损失永久坏死 | AI 的失败方式是否低级到污染体验判断 |
| 胜负可读性 | runtime 可验证状态转换、对象清理、关键 HUD 状态不崩 | 玩家是否能说清楚“我为什么赢 / 输 / 卡住”，而不是靠猜 |

结论：客观证据只能说明“值得拿给用户判断”；不能自动把 `M4` 标成已通过。

## 3. 用户短局试玩应看什么

建议用户只打一局短 Alpha，不需要追求完整内容体验。重点观察：

- 开局是否能理解：基地、工人、资源点、敌我关系是否清楚。
- 前几分钟是否有目标：采集、补工人、出兵、防守或进攻是否自然出现。
- 中期是否有压力：AI 是否会推进、恢复或制造威胁，而不是静止或第一波后坏死。
- 控制是否拖后腿：选择、右键、队列、建造、训练、集结、HUD 是否让玩家觉得自己被系统卡住。
- 防守和反打是否存在：玩家是否还有补救空间，而不是进入某个不可逆锁死状态。
- 结束或卡点是否可解释：赢、输、僵持、卡死分别能不能说出原因。
- 失败是否有用：如果不通过，失败原因应能落到控制、AI、节奏、胜负表达或具体工程合同上。

## 4. M4 通过 / 带债通过 / 失败

| 结论 | 判定标准 | 后续处理 |
| --- | --- | --- |
| `Alpha 通过` | 一局短人机 Alpha 有开始、中期压力、可理解的结束或结果；控制没有明显不公平；AI 没有低级坏死；用户能解释胜负或卡点原因 | 可以把 `M4` 记为人工通过，并把剩余小问题转成后续 polish 或 hardening |
| `带债通过` | 核心 loop 成立，但存在不阻断判断的问题：节奏偏粗、反馈不够清楚、AI 压力不稳定、胜负表达仍需要加强 | 记录债务，不阻止进入后续里程碑；必须明确哪些债务会影响 M5/M6 |
| `失败：控制问题` | 玩家因为选择、命令、HUD、队列、建造或移动反馈无法公平操作 | 回到工程合同，补复现和 regression，不作为品味争论 |
| `失败：AI 问题` | AI 第一波后永久卡死、经济恢复失败、出兵停滞，或行为低级到无法评价对局 | 回到 AI economy / recovery contract |
| `失败：节奏问题` | 有功能但没有 match loop：没有中期压力、无限拖延、玩家不知道该推进什么 | 先拆成可测节奏证据和 AI/目标任务，再讨论产品味道 |
| `失败：胜负不清晰` | 玩家赢了、输了或卡住了，但无法理解原因；结束状态缺失或误导 | 回到状态表达、HUD、对局结束合同 |

`带债通过` 不能变成“什么都算过”。它只适用于用户已经能判断 match loop 成立，但能清楚列出后续债务的情况。

## 5. 应回工程合同而不是争论品味的问题

以下问题应直接转成 engineering contract / test task：

- 选中失败、右键命令不生效、命令队列丢失或 interrupt / restore 行为不一致。
- 资源、供给、采矿饱和、训练成本、单位出生、建造取消 / 中断 / 续建出现可复现错误。
- AI 工人损失后不恢复、建筑损失后不重建、第一波后永久停摆、impossible placement spam 等恢复问题。
- 对局状态、死亡清理、对象引用、占地释放、路径阻塞导致不可恢复卡死。
- HUD 状态错误、命令卡误导、胜负状态没有表达或表达与实际不一致。
- 同一操作在相同初始状态下结果不稳定，无法靠用户“感觉”判断。

这些不是 taste。只要能复现、能定义期望行为、能写 regression，就应回工程合同处理。

## 6. 用户回来前 Codex / GLM 可以继续什么

在用户返回前，Codex/GLM 可以继续准备 M4 的客观包，但不能声称 `M4` 已人工通过：

- 维持并补齐 `m4-match-loop`、`m4-ai-recovery`、`ai-economy`、`m4-player-reported-issues` 等 regression 证据。
- 跑并记录 build、typecheck、相关 runtime packs 和本地残留清理结果。
- 把可复现的控制、AI、节奏、胜负表达问题拆成工程合同任务。
- 准备短局试玩脚本，让用户回来后能快速判断，而不是被实现细节拖住。
- 继续推进不冲突的 M5/M6/M7 客观准备，但保持 `M4` 状态为“客观材料准备中 / 待用户人工裁决”。
- 不让 GLM 代替用户判断“是否像一局 Alpha”，也不把测试通过等同于人工验收通过。
