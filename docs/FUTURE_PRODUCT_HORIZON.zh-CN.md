# 未来产品地平线：M0-M7 之后我们到底要去哪里

> 用途：把项目的视角从“当前施工段 `M0-M7`”拉回到“整个产品从现在到终局还要穿过哪些地平线”。  
> 结论先写在最前：`M0-M7` 不是项目未来，它只是当前这段施工的 gate。

## 1. 为什么必须把 `M0-M7` 降级

`M0-M7` 很有用，但它有一个天然问题：

```text
它描述的是当前团队正在切哪几段路，
不是整个产品从 alpha 基底走到可长期游玩的完整未来。
```

如果继续用 `M0-M7` 当主镜头，就会持续出现三个偏差：

1. 看起来一直很忙，但视野还是停在眼前施工
2. 未来会被误写成“当前里程碑的延长线”
3. 角色计划会被当前 gate 拖窄，而不是服务长期终态

所以从现在开始要分成两层看：

- `施工层`：`M0-M7`
- `产品未来层`：未来产品地平线

而“未来产品层”具体由哪些能力构成，不在本文件里展开，统一见：

- `WAR3_PAGE_PRODUCT_MASTER_PLAN.zh-CN.md`
- `WAR3_MASTER_CAPABILITY_PROGRAM.zh-CN.md`

## 2. 未来产品地平线总览

| 地平线 | 名称 | 真正目标 | 当前位置 |
| --- | --- | --- | --- |
| `H0` | 产品壳层地平线 | 让项目先成为“完整页面版产品”，而不是直接进局的内核 | 明显缺失 |
| `H1` | RTS 合法性地平线 | 让项目摆脱“网页样机”感，成为可信 RTS alpha 基底 | 正在穿越 |
| `H2` | 战场识别地平线 | 让第一眼开始像 War3-like 战场 | 即将主攻 |
| `H3` | 短局闭环地平线 | 让 10-15 分钟人机短局成立 | 未通过 |
| `H4` | 战略骨架地平线 | 让项目开始有 timing / tech / composition / counter | 未来主轴 |
| `H5` | 身份系统地平线 | 让项目具备 War3-like 的识别性身份层 | 更后期 |
| `H6` | Beta 产品地平线 | 在选定范围内做到 content-complete 和可持续验证 | 更后期 |
| `H7` | 外部候选地平线 | 能私测、能公开、能对外收反馈 | 更后期 |
| `H8` | 长期产品地平线 | patch、扩展、长期演化 | 终局后 |

## 3. 这 9 条地平线分别在解决什么

## `H0` 产品壳层地平线

它解决的是：

- 主菜单
- 模式选择
- 对局前配置
- loading / briefing
- pause / results / return flow
- settings / help

换句话说：

```text
项目先要从“打开页面直接进局的原型”
变成“一个有前门、有会话控制、有收口的完整页面版产品”。
```

对应关系：

- 主需求：完整页面版产品结构
- 版本：跨 `V1-V8` 的横切层
- 当前没有正式 gate，但它不能再被当成后期装饰
- 顶层定义：`WAR3_PAGE_PRODUCT_MASTER_PLAN.zh-CN.md`

## `H1` RTS 合法性地平线

它解决的是：

- 命令和控制权可信
- 经济/建造/生产可信
- 战斗/单位存在/路径可信
- 硬化和重构不再持续倒退

换句话说：

```text
玩家至少愿意承认“这已经是一款 RTS alpha 的地基”。
```

对应关系：

- 主需求：`D1`、`D2`、`D3`、`D9`
- 大山：`G1`
- 版本：`V1-V2.5`
- 当前主要 gate：`M0-M2`、`M7`

## `H2` 战场识别地平线

它解决的是：

- TH/矿/树线/出口有没有基地语法
- worker/footman/建筑/资源点是否一眼能读
- 镜头、HUD、占地提示是不是在服务 RTS，而不是破坏它

换句话说：

```text
玩家第一眼开始觉得“这像一个 War3-like 战场”，
而不是一组通过测试的对象。
```

对应关系：

- 主需求：`D4`
- 大山：`G2`
- 版本：`V3.1-V3.4`
- 当前主要 gate：`M3`

## `H3` 短局闭环地平线

它解决的是：

- 一局 10-15 分钟是否成立
- AI 是否能形成压力和恢复
- 玩家是否能防守、补兵、反打
- 赢/输/卡住是否清楚

换句话说：

```text
项目从“可信 opening”长成“可信短局 alpha”。
```

对应关系：

- 主需求：`D5`
- 大山：`G3`
- 版本：`V4.1-V4.4`
- 当前主要 gate：`M4`

## `H4` 战略骨架地平线

它解决的是：

- production / queue / rally / repair
- tech / prerequisite / upgrade
- roster 扩张
- composition / timing / counter

换句话说：

```text
项目开始从最小循环长成真正有战略选择的 RTS。
```

对应关系：

- 主需求：`D6`
- 大山：`G4`
- 版本：`V5.1-V5.4`
- 当前主要 gate：`M5` 只是方向决定，不是完成标志

## `H5` 身份系统地平线

它解决的是：

- hero / ability
- neutral / creeps / items
- faction asymmetry
- 项目识别度

换句话说：

```text
项目不只是“像 RTS”，而是开始具备 War3-like 的身份层。
```

对应关系：

- 主需求：`D7`
- 大山：`G5`
- 版本：`V6.1-V6.4`
- `M5` 只决定做不做，不代表做完

## `H6` Beta 产品地平线

它解决的是：

- 选定范围 content-complete
- 平衡、bugfix、性能、可用性
- 外部 tester 不需要内部语境也能理解

换句话说：

```text
项目从“方向正确的 alpha”进入“可连续磨的 beta 候选”。
```

对应关系：

- 主需求：`D6-D8`
- 版本：`V7.1-V7.3`
- 当前没有对应现成里程碑，未来需要新 gate

## `H7` 外部候选地平线

它解决的是：

- private playtest
- public demo
- README / Known Issues / disclosure
- feedback intake / triage

换句话说：

```text
项目开始能面向外部世界，而不是只在内部自洽。
```

对应关系：

- 主需求：`D8`
- 大山：`G6`
- 版本：`V8.1-V8.3`
- `M6` 只相当于很早的候选 gate，不是整个外部产品化工作

## `H8` 长期产品地平线

它解决的是：

- patch / hotfix
- expansion
- 长期路线
- 产品生命期管理

换句话说：

```text
项目不再只是一个阶段性 demo，
而是一个会继续活下去的产品。
```

对应关系：

- 版本：`V9.1-V9.3`
- 当前没有对应现成里程碑

## 4. `M0-M7` 在未来里到底占多大

正确说法应该是：

| 施工 gate | 实际覆盖 |
| --- | --- |
| `M0-M2` | `H1` 的前半段 |
| `M3` | `H2` 的第一个正式 gate |
| `M4` | `H3` 的第一个正式 gate |
| `M5` | `H4-H5` 的方向选择 gate，不是完成 gate |
| `M6` | `H7` 的早期候选 gate，不是完整外部产品化 |
| `M7` | `H1` 尾部的 hardening gate |

所以必须明确：

```text
M0-M7 只覆盖了产品未来前部的一小段。
它不是未来本身。
```

## 5. 以后角色计划应该怎么写

以后每个角色计划都必须分成两层：

### 第一层：未来地平线职责

这个角色在 `H0-H8` 各自承担什么。

### 第二层：当前施工 gate

这个角色在 `M0-M7` 这段里当前手上要做什么。

如果一个角色计划只有 `M0-M7`，就说明它还是被当前施工视角绑住了。

## 6. 当前最该纠正的思维方式

以后不要再用下面这种句式：

```text
做完 M7，我们项目就进入后期了。
```

正确句式应该是：

```text
做完 M7，我们只是把 H1 的尾部收干净；
H0 这条产品壳层线还要补；
接下来才真正主攻 H2，随后还要穿过 H3、H4、H5、H6、H7、H8。
```

## 7. 当前真实位置

当前更准确的口径是：

```text
H0 的最小产品壳层明显缺失；
我们正在 H1 尾部，
准备正式进入 H2，
H3 还没过，
H4-H8 大部分还在未来。
```

这才是“未来”视角，而不是“当前施工视角”。
