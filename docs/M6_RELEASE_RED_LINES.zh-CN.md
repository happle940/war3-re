# M6 Release Red Lines

> 用途：定义 `M6` 阶段哪些问题会直接阻止外部分享，哪些问题只允许内部或少量私测。
> 原则：先挡住会污染反馈的风险，再讨论“能不能发”。

---

## 1. 判断等级

`M6` 只分三档：

- `禁止任何外部分享`
- `可少量私测，不可公开`
- `可讨论公开分享`

默认从最保守档开始往上抬，不反过来。

---

## 2. 一级红线：禁止任何外部分享

满足任一项，就不能发给外部 tester，包括少量私测。

- 页面打不开、白屏、启动崩溃，或依赖隐藏本地步骤
- 初始 HUD 不出现，或核心对象无法识别
- worker 不能稳定选择、移动、采集
- Town Hall / Barracks 命令卡不可读，最小训练闭环断裂
- 资源扣除、单位出生、采集循环存在明显破坏性错误
- AI 完全不行动，或对局表现为“玩家独自在空地图操作”
- 存在明显不可恢复卡死、持续刷错、严重性能故障
- 已知问题文档缺失或明显落后于真实状态
- 分享入口没有说明 alpha 范围、proxy 视觉、当前非目标
- 用户尚未批准 release，但对外措辞已经把当前版本包装成可试玩 demo

这些问题由 Codex/GLM 继续收工程证据，不允许拿外部反馈来“顺便测一下”。

---

## 3. 二级红线：可少量私测，不可公开

下面问题存在时，可以考虑受控私测，但不应公开扩散。

- `M3` 的镜头 / 空间感还没人工确认
- `M4` 的人机 alpha 还没完成人工 judgment
- `M5` 的内容与视觉方向仍未最终确认
- 胜负表达仍然偏原型，容易让陌生 tester 误判项目成熟度
- README / 分享说明仍不足以让陌生人独立理解项目边界
- known issues 虽已存在，但仍有多项会强烈影响第一印象的问题
- live smoke path 文档已写好，但目标版本还没有留下执行记录

这个档位的私测必须满足：

- tester 数量小
- tester 知道这是 gameplay alpha
- 反馈目标被明确限定
- 收集的是“理解/操作/节奏”反馈，不是“完成度”评分

---

## 4. 哪些债可接受为少量私测债

下面这些问题如果被明确披露，可以不自动阻止少量私测：

- proxy / fallback 视觉仍较粗
- 产品方向尚未最终决定
- 未包含完整阵营、完整科技树、完整 onboarding
- 文本包装、README、分享页还不够像正式产品
- M7 工程硬化仍在进行，但当前可玩路径已有合同保护

这些债的前提是：不会让 tester 连“这是什么、该怎么操作、当前要验证什么”都搞不清。

---

## 5. 每类红线谁负责

| 问题类型 | 默认 owner | 备注 |
| --- | --- | --- |
| 启动 / build / typecheck / smoke 崩溃 | Codex + GLM | 先修客观问题 |
| HUD / 选择 / 命令卡 / 采集 / 训练断裂 | Codex + GLM | 先回合同与 regression |
| AI 像摆设、恢复失效、节奏断裂 | Codex + GLM | 先做 M4/M6 客观包 |
| alpha 范围说明不清 | Codex | 文档边界问题 |
| 是否允许私测 / 公开分享 | 用户 | Codex 只能准备输入，不能代批 |

---

## 6. 当前默认结论

以当前材料状态，默认仍应视为：

`可继续准备私测输入，但还不能声称可公开，也不能默认私测已批准。`

更直接地说：

- 工程和文档可以继续朝 `少量私测` 准备
- 但只要一级红线未被重新核验并关闭，就不要往外发

---

## 7. 复核规则

每次出现新的候选版本，都要重看这 4 个来源：

- [M6 Release Brief](./M6_RELEASE_BRIEF.zh-CN.md)
- [M6 Live Smoke Path](./M6_LIVE_SMOKE_PATH.zh-CN.md)
- [Known Issues](./KNOWN_ISSUES.zh-CN.md)
- 当前候选版本的 build / typecheck / smoke 结果

如果 red line 判断靠“记得之前好像测过”，就等于还没过线。
