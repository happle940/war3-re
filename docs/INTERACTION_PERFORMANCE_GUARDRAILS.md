# War3 RE - 长期交互与性能护栏

## 1. 这份文档解决什么问题

`ROADMAP_STRATEGY.md` 解决的是产品路线和阶段目标。  
`GLM51_EXECUTION_GUIDE.md` 解决的是当前原型如何进入可持续开发。

这份文档解决第三个问题：

> 当项目继续演进到“网页版完整 war3 体验”时，  
> 现在的每一个交互和架构决策，应该提前服从哪些长期约束。

它的目的不是让 `glm` 一次性实现完整 war3，  
而是避免它在每个 cycle 只盯着眼前功能，把未来的完整交互和性能空间提前堵死。

---

## 2. 最终目标怎么定义

这里说的“完整网页版 war3”，不是指把所有内容立刻做完，  
而是指系统设计上要能够容纳以下完整体验：

- 单人 1v1 对战可完整打完一局
- 人族、兽族、不死、暗夜精灵都可以被接入
- 英雄、技能、物品、商店、升级、迷雾、控制组、快捷键全部成立
- 小地图、编队、右键智能命令、目标选择、技能指示、建筑放置都符合 RTS 直觉
- 在浏览器里保持可玩的性能底线

这意味着从现在开始，后续所有实现都必须回答两个问题：

1. 这个实现能不能自然扩展到“完整 war3 交互”？
2. 这个实现会不会在“单位更多、技能更多、UI 更复杂”时把浏览器拖死？

---

## 3. 不可违反的长期原则

这些原则从现在开始就应该成立，否则越往后代价越大。

### 3.1 模拟真相不能依赖 Three.js 对象

Three.js 负责表现，不负责游戏真相。

长期必须形成下面这个关系：

- 游戏状态是权威数据
- 渲染对象只是状态的投影
- 输入不会直接改 mesh
- AI 不会直接改 mesh
- UI 不会直接改 mesh

短期内允许保留一部分 `mesh.position` 依赖，  
但新功能不能继续加深这种耦合。

### 3.2 所有玩家交互最终都要走命令层

完整 war3 的所有关键交互，最终都必须能映射成统一命令。

至少包括：

- move
- attack
- attack-move
- stop
- hold-position
- patrol
- gather
- return-resources
- build
- train
- set-rally
- cast-ability
- use-item
- buy-item
- revive-hero

原因不是“为了优雅”，而是为了以后这些入口能共用：

- 玩家鼠标点击
- 快捷键
- AI
- 脚本事件
- 回放
- 未来如果要做联机，命令序列也更容易抽出来

### 3.3 交互语义必须先于 UI 呈现

先定义“行为语义”，再做“按钮和热键”。

例如：

- 右键敌人到底是 attack 还是 move？
- 右键金矿是 gather 还是 move？
- 圣骑士点神圣之光后，点击己方英雄、点击敌方亡灵、点空地各是什么意思？
- 选中多个建筑时，Tab 如何切换子分组？

这些都应该先在交互模型里成立，  
再映射到命令卡、鼠标、热键和小地图。

### 3.4 UI 必须是数据投影，不是逻辑源头

命令卡、训练队列、物品栏、英雄技能栏、小地图提示，都只能是状态投影。

不能形成这种结构：

- 点了某个按钮，按钮内部自己决定复杂逻辑
- UI 自己维护一套和游戏状态不同步的局部真相

长期正确结构应该是：

- 游戏状态决定 UI 展示
- UI 触发命令
- 命令改变游戏状态
- 状态再反映回 UI

### 3.5 近邻查询不能永远靠全表扫描

完整 war3 里以下系统都会依赖空间查询：

- 自动仇恨
- 近战/远程攻击索敌
- 视野
- 碰撞分离
- 框选
- 技能范围判定
- 野怪仇恨
- 迷雾更新

所以项目必须逐步过渡到统一空间索引，例如：

- spatial hash
- uniform grid
- 分区桶

可以不马上做，但绝不能继续默认所有交互都用 `for units -> for other units`。

### 3.6 路径与占用必须共享同一张地图语义

完整 war3 的交互会依赖统一的地图运行时语义：

- 哪里可走
- 哪里被建筑占用
- 哪里是树
- 哪里是矿
- 哪里是浅水/深水/悬崖
- 哪些单位能通过、哪些不能

这意味着不能把这些信息分散在：

- 地形渲染器
- 场景里的树模型
- 建筑 Mesh
- 选择逻辑

长期必须有一个统一的 map runtime / occupancy / pathing truth。

### 3.7 关键系统必须允许降频运行

在浏览器里，不是所有系统都该每帧跑。

长期上至少这些系统应该允许独立 tick：

- AI 决策
- 迷雾更新
- 路径重规划
- 小地图重绘
- 经济决策
- 部分 UI 刷新

如果这些全都锁死在每帧运行，后面复杂度一上来就会撞墙。

## 4. 完整 war3 交互模型应该长成什么样

下面不是要求立刻实现，而是要求以后新功能不要违背这个方向。

### 4.1 选择模型

最终必须支持：

- 单选
- 框选
- 双击同类
- Ctrl + 数字编队
- 数字快速选组
- 多建筑选择
- Tab 子分组切换
- 英雄优先显示

对架构的要求：

- 选择状态必须独立于渲染对象
- 选择分组需要有稳定的数据结构
- UI 不应该自己猜测“当前主单位是谁”，而应该由 selection model 明确给出

### 4.2 智能右键语义

完整 war3 的右键不是“移动键”，而是上下文智能命令。

最终至少要支持：

- 右键地面：move
- 右键敌人：attack
- 右键金矿：gather gold
- 右键树区：gather lumber
- 右键己方受伤机械单位：repair（后期）
- 右键商店：交互 / 购买
- 右键可进驻目标：enter / load（后期）

所以右键系统的长期正确结构是：

1. 识别点击目标语义
2. 根据选中单位类型判断合法交互
3. 生成具体命令
4. 再交给命令执行层

### 4.3 技能与目标指示

完整 war3 最复杂的交互之一就是技能目标模式。

最终必须支持的目标类型：

- 无目标技能
- 单位目标技能
- 地点目标技能
- 范围目标技能
- 切换类技能
- 物品技能

这意味着后面必须有独立的 interaction state：

- 当前是否处于“释放技能模式”
- 鼠标悬停目标是否合法
- 点击左键是确认施法还是普通选择
- 右键是取消施法还是下达右键命令

所以千万不能把技能交互继续塞成“按钮 onClick 直接扣蓝并生效”。

### 4.4 建筑交互

完整 war3 的建筑交互不是只有“点一下训练”：

- 建筑放置预览
- 占地校验
- 遮挡校验
- 是否贴矿 / 贴树 / 卡住入口
- rally point
- 多建筑生产
- 队列显示
- 升级与取消

所以建筑系统长期需要支持：

- 放置验证
- 生产队列
- rally 命令
- 建筑命令卡
- 多建筑共享交互语义

### 4.5 英雄交互

英雄会把整个交互复杂度拉高一个层级。

完整体验至少包括：

- 技能升级
- 技能施法
- 物品栏
- 物品使用 / 拖动 / 丢弃 / 购买
- 复活
- 经验 / 等级 / 属性展示
- 英雄优先子组

这意味着不能把英雄当成“多几个数值的普通单位”。

长期必须至少预留：

- ability bar
- hero inventory
- hero progression model
- hero-only interaction priority

### 4.6 小地图交互

完整 war3 小地图不仅是展示。

最终必须支持：

- 点击跳视角
- 右键在小地图上发命令
- 迷雾呈现
- 友军 / 敌军 / 建筑 / 资源点标记
- ping / 提示（后期）

所以小地图长期应该是一个独立交互表面，不只是每帧重画一个贴图。

### 4.7 热键体系

最终热键不只是按钮快捷入口，而是完整输入语言。

至少包括：

- A 攻击
- S 停止
- H 原地待命
- P 巡逻
- B 建造子菜单
- QWER/ZXCV 类命令卡热键
- 数字编队
- Tab 子组切换
- Esc 取消当前模式

因此输入系统必须逐步走向：

- 输入路由层
- 模式状态
- 热键映射
- 命令发射

而不是继续在 DOM 按钮点击和鼠标事件里散着写逻辑。

## 5. 浏览器版本的性能底线

目标不是“Web 和原生引擎一样奢侈”，  
而是“在目标硬件上稳定可玩”。

### 5.1 最终性能目标

对于完整网页版 war3，建议把性能目标写死为：

- 目标硬件：普通中档笔记本浏览器
- 目标帧率：混战时稳定 30fps 以上
- 目标地图：128x128 为主，必要时更大地图通过降级策略处理
- 目标可见单位：200 左右可见单位时仍然可玩

不是所有情况下都追 60fps。  
对 RTS 来说，稳定和输入响应比绝对高帧更重要。

### 5.2 33ms 帧预算建议

建议长期按这个预算思考：

- 渲染：12-16ms
- 游戏模拟：8-10ms
- 路径/占用/空间查询：3-5ms
- UI / 小地图 / HUD：2-4ms
- 安全余量：2-4ms

### 5.3 立即触发性能治理的红线

只要出现下面任一项，就暂停堆功能，先治性能：

- 100 单位场景低于 50fps
- 200 单位场景低于 30fps
- 明显输入延迟或镜头拖泥带水
- 新增一个系统后帧率断崖下降
- 连续玩 10 分钟后帧率明显下降
- 地图切换后内存持续上涨

## 6. 哪些实现方式以后一定会出问题

下面这些现在可能还能跑，但如果一直这么做，完整 war3 阶段一定会出问题。

### 6.1 每帧全表扫描索敌

现在原型阶段能接受，完整阶段不行。

以后需要过渡到：

- 空间索引查询
- 分层仇恨更新
- 降频索敌

### 6.2 每个单位都挂很多独立 3D UI 对象

单位血条、状态图标、选中圈、目标指示如果都继续一个单位一个对象无限堆，
后面 draw calls 和 scene graph 开销会逐步失控。

长期要考虑：

- 复用
- pooling
- 批渲染
- 或把部分 3D UI 改成更轻的绘制方式

### 6.3 每帧重建整个 UI

命令卡、训练队列、物品栏、英雄栏、小地图，如果都每帧全量重建，
随着交互复杂度上升会变成持续负担。

长期应改为：

- 状态变更驱动更新
- 脏标记
- 局部刷新

### 6.4 每个命令都立即直改单位状态

这会让：

- 玩家输入
- AI
- 热键
- 技能
- 商店
- 小地图命令

最后都绕成一团。

长期要统一到命令入口，再从命令映射到单位状态机。

### 6.5 寻路每帧重算

这是浏览器 RTS 最容易炸掉的点之一。

正确方向应该是：

- 收到命令时求路
- 阻挡变化时再重规划
- 局部失效才重算
- 群体移动共享路径信息

## 7. 建议的长期落地顺序

后续 cycle 不要随意跳，建议按这个顺序推进。

### Phase A：交互骨架

目标：

- 命令层
- 选择模型
- 输入模式状态

建议顺序：

1. 统一命令入口
2. 把右键、建造、训练全部走命令层
3. 建立 selection model
4. 建立基础热键入口

### Phase B：地图与模拟骨架

目标：

- occupancy / blockers
- pathing
- 空间索引
- 固定 tick / 降频系统

建议顺序：

1. blocker 与占用语义统一
2. A* 与重规划频率控制
3. spatial hash / grid
4. AI / 迷雾 / 小地图独立 tick

### Phase C：完整 RTS 交互

目标：

- control groups
- rally
- attack-move / hold / patrol
- 小地图命令
- 多建筑与子组

### Phase D：英雄与复杂交互

目标：

- 技能目标模式
- 物品栏
- 商店
- 英雄复活
- 英雄升级和属性交互

### Phase E：高复杂度系统

目标：

- 战争迷雾
- 中立建筑
- 野怪
- 多种族复杂技能
- 更高单位量下的性能治理

## 8. 对 GLM 的长期约束

从现在开始，凡是做交互相关功能，`glm` 都必须先回答这几个问题再写代码：

1. 这个交互最终对应哪种命令？
2. 未来热键入口是否能复用这套逻辑？
3. 未来 AI 是否能通过同一入口触发？
4. 这个实现会不会让 UI 成为逻辑源头？
5. 这个实现会不会让完整 war3 阶段的性能变差很多？

如果这 5 个问题里有 2 个以上答不上来，就说明设计还不够成熟。

## 9. 以后给 GLM 的交互类 prompt 必须带的补充说明

后续只要是命令层、快捷键、选择、技能、英雄、物品、小地图相关任务，
都建议附带下面这段：

```md
长期约束：

这不是只服务当前原型的临时实现，而是要服务“网页版完整 war3”。
请在实现时同时考虑未来这些系统的可接入性：

- 快捷键
- 控制组
- 小地图交互
- 英雄技能目标模式
- 物品栏与商店
- AI 通过同一入口发命令
- 战争迷雾与空间查询

因此请遵守：
1. 新交互优先映射成统一命令，而不是直接在输入事件里散改状态
2. UI 只能作为状态投影和命令入口，不能成为逻辑真相
3. 不要做只对当前 demo 有效、以后会卡死完整 war3 的实现
4. 如果你采用的是“先简化版实现”，必须明确说明未来如何自然扩展到完整 war3
```

## 10. 当前对你最重要的一句话

你现在不是在做一个“眼前能玩的 demo”。  
你是在给“浏览器里的完整 war3”铺一条不会越走越窄的路。

所以接下来的每一轮都应该追求：

> 局部实现可以小，  
> 但方向必须能容纳完整 war3 的交互与性能复杂度。
