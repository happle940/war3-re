# M6 Private Playtest Feedback Triage：私测反馈分流

> 用途：把少量私测反馈转成可执行判断。本文不批准私测或公开分享，只定义收到反馈后如何按严重度、合同类型和用户判断边界处理。

## 1. Feedback Buckets 优先级

按下面顺序处理反馈，不要先讨论品味或内容愿望。

| 优先级 | Bucket | 判断标准 | 默认动作 |
| --- | --- | --- | --- |
| P0 | 停止外发 / red line | 页面打不开、白屏、启动崩溃、核心对象不可见、worker 无法选中或移动/采集、训练闭环断裂、AI 完全不动、不可恢复卡死、严重性能或持续刷错。 | 立即 `hold`；停止继续发私测链接；转工程 owner。 |
| P1 | 私测阻断但可内部复验 | 会污染少量 tester 反馈，例如 HUD/点击/镜头严重误导、Known Issues 过期、私测说明缺失、反馈路径不清。 | 暂停扩大私测；补证据、文档或 focused fix 后再判断。 |
| P2 | 工程合同缺口 | 可复现、能写期望行为、能用 runtime/spec/smoke 证明，例如资源扣除异常、单位出生错误、stale HUD、AI 活动状态不一致。 | 转 regression / contract task；必要时加入候选阻断。 |
| P3 | 私测有效债 | 不阻断受控私测，但会影响体验理解，例如 proxy 视觉粗糙、缺重开按钮、赛后总结不足、内容少、README 不够像正式入口。 | 记录为 debt；私测说明中披露；不冒充已解决。 |
| P4 | 用户判断项 | “空间感够不够”“人机节奏是否有趣”“是否值得公开”“M5 方向怎么选”等无法只靠工程合同关闭的问题。 | 记录到 deferred judgment；等待用户裁决。 |
| P5 | 无效 / 重复 / 已覆盖反馈 | 已在 Known Issues 披露、已由 runtime 合同覆盖，或 tester 评价的是最终美术/完整内容但当前不测该目标。 | 归档为 context note；不进入工程队列。 |

## 2. Must-fix Before Wider Sharing

下面问题在更大范围分享前必须先修或至少转为明确 `hold`：

- 启动不可用：白屏、崩溃、隐藏本地步骤、部署链接不稳定。
- 初始可读性断裂：HUD 不出现，默认镜头下 Town Hall、worker、金矿等核心对象不可见。
- 核心控制断裂：worker 不能选中，右键移动 / 采集没有反馈，命令卡不可读。
- 生产闭环断裂：Town Hall / Barracks 无法训练，资源扣除或单位出生明显错误。
- AI 存活性失败：AI 完全不采集、不生产、不行动，tester 像在空地图操作。
- 反馈污染：HUD、点击、镜头、持续错误或性能问题让 tester 无法给出有效反馈。
- 外部说明失败：Known Issues 过期，私测邀请没有说明 alpha 范围、proxy 视觉、非目标和反馈边界。
- 授权边界失败：用户未允许外发，但消息或文档把链接包装成公开 demo。

这些问题不是“品味问题”。它们应直接回到工程、文档或 release gate。

## 3. 可记录为 Debt 但不阻断当前私测 lane

如果已提前披露，下面问题可以记录为债务，不自动阻断少量知情私测：

- 视觉仍是 proxy / fallback，不代表最终美术。
- 单位、建筑、阵营、科技树和战役内容仍不完整。
- 胜负状态已表达但缺少重开按钮、赛后统计或发行级包装。
- `M3` 空间 / 镜头 / 视觉方向、`M4` 人机 Alpha 质量、`M5` 产品方向仍待用户判断。
- README / share copy 还不是正式公开入口，只能支撑少量私测上下文。
- AI 压力“有点弱 / 有点强”但 AI 不是静止样机，且没有破坏性合同失败。
- 长期平衡、正式 onboarding、最终资产质量、公开传播适配度。

这些债可以影响 `public share` 判断，但不应把当前工程 lane 打断成泛泛产品重做。

## 4. 工程合同失败 vs Taste Feedback

把反馈先问成三个问题：

1. 能否复现？
2. 能否写出期望行为？
3. 能否用测试、smoke、截图/录屏或日志证明？

如果三项都能回答，优先按工程合同处理。

工程合同失败示例：

- “worker 点了金矿但没有采集”，且能给出浏览器、步骤、截图或录屏。
- “训练 footman 后资源扣了但单位没出生。”
- “AI 10 分钟完全没有采集或出兵。”
- “切换选择后命令卡显示旧按钮。”
- “页面打开后持续刷 console error 并卡死。”

Taste / 用户判断示例：

- “这个 proxy 不像最终 Warcraft-like。”
- “镜头感觉不够舒服”，但核心对象可见、HUD 不遮挡、操作能完成。
- “AI 压力不够好玩”，但 AI 能采集和生产。
- “内容量太少，不像完整 demo。”
- “我觉得应该先做 Orc。”

混合反馈要拆开：可复现的工程部分进合同；方向、品味、内容范围留给用户。

## 5. Codex 转译规则：Queue / Regression / Defer

Codex 收到一条反馈后，按下面方式转译，不直接把原话丢进任务队列。

| 反馈状态 | 转译目标 | 处理方式 |
| --- | --- | --- |
| P0 red line | `hold` + 工程修复候选 | 写最小复现、环境、阻断级别、owner；先不扩大私测。 |
| 可复现合同失败 | regression / contract task | 写期望行为、当前行为、最小复现、建议 spec/smoke proof。 |
| 证据不足但可能严重 | 复验 bucket | 要求补浏览器、版本、截图/录屏、console、候选链接；暂不当 bug 关闭或确认。 |
| 文档/上下文误导 | docs/share-copy bucket | 更新私测说明、Known Issues、反馈模板或 stop-and-report 文案；不改代码。 |
| 已披露可接受债 | debt bucket | 记录为已知限制；确认 tester 已知情；不阻断当前私测 lane。 |
| 方向/品味判断 | defer to user | 放入用户待裁决材料；不要让 GLM/Codex 代判。 |
| 已覆盖/重复 | archive | 标记为重复或已覆盖；必要时指向 Known Issues 或已有合同。 |

写工程任务时必须避免：

- “修一下手感。”
- “让它更像成品。”
- “AI 更聪明一点。”
- “HUD 更好看。”

应改写为：

```text
期望行为：
当前行为：
最小复现：
证据：
阻断级别：
建议 proof：
默认 owner：
```

## 7. Triage 口径

- 私测反馈能帮助判断候选版本，不等于用户已批准 M6。
- 一条严重反馈应先问“是否污染继续私测”，再问“是否影响公开分享”。
- 没有证据的严重反馈不能直接关闭，也不能直接当作已确认 bug；先补复验。
- 已披露债务不应反复阻断当前 lane，但如果 tester 大量误解同一债务，说明文档/邀请说明需要修。
- 任何会改变产品方向、视觉身份或公开发布策略的判断，都必须回到用户。
