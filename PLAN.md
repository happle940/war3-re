# War3 RE - 动态总控规划

> 最后更新：2026-04-09
> 当前状态：Cycle 3E 后
> 文档定位：这是项目的总控计划，不是某一个阶段的功能清单。
> 它需要随着项目现实不断调整，优先回答三个问题：
> 1. 我们现在到底处在什么阶段。
> 2. 当前最大的矛盾是什么。
> 3. 接下来什么最值得做，什么暂时不该做。

---

## 1. 项目定义

### 1.1 北极星目标

在浏览器里做出一个让 war3 老玩家愿意打完一局、并且打完后会说“这就是 war3 的感觉”的单人 1v1 RTS。

这个目标包含 4 个同时成立的维度：

1. 玩法语义正确
- 采集、建造、训练、战斗、侦察、英雄、科技树这些核心机制必须成立。

2. 玩家感知接近 war3
- 镜头、比例、地图语法、反馈、HUD、节奏，必须让老玩家感到熟悉。

3. 浏览器里可玩
- 不是“能跑”，而是“能玩”。
- 目标性能底线：中等场面稳定 30fps+。

4. 架构可持续扩展
- 不能靠不断堆 if/else 和一个大文件把项目做死。
- 未来必须能容纳 4 种族、英雄、物品、迷雾、AI、快捷键、控制组等完整 war3 交互。

### 1.2 当前最准确的项目判断

这个项目现在不是“war3 复刻版”，也不是“接近完成的 alpha”。

它当前最准确的定位是：

> 一个已经具备较完整 RTS 运行时骨架的系统原型。

也就是说：

- 系统层已经比纯 demo 前进了很多
- 但玩家感知层还远远没有到 “像 war3”
- 现在最大的缺口已经从“有没有系统”转移为“系统和感知有没有被统一到同一个产品目标上”

## 2. 当前真实状态

### 2.1 已建立的主干能力

截至 Cycle 3E，仓库已经具备这些真实能力：

- `MapRuntime`：统一地图查询接口
- `TeamResources`：按 team 独立资源管理
- `GamePhase`：基础游戏阶段状态
- `GameCommand`：统一命令入口基础版
- `SimpleAI`：低频 tick AI
- `OccupancyGrid`：建筑占用与放置校验
- `PathingGrid`：统一 blocker 查询
- `PathFinder`：单单位 A* 寻路基础
- `TreeManager`：树木 blocker 与资源语义
- `ResourceTarget`：gold/tree 明确资源目标
- W3X 地图加载、地形渲染、小地图、基础 HUD

这些说明：

> 项目已经跨过“完全无结构的原型期”，进入“可以认真规划产品形态”的阶段。

### 2.2 当前最严重的问题不再是“缺系统”，而是“缺 war3 感”

从当前可运行版本看，系统主干已经开始成型，但玩家第一眼看到的仍然是：

- 镜头像测试板，不像 war3
- 地图过于平、空、均匀，没有空间语法
- 单位/建筑/地图比例关系不对
- 视觉材质、光照、氛围几乎没有建立
- HUD 是功能性面板，不是 war3 风格的信息框架

所以当前最大的矛盾不是：

- “要不要再补一个系统”

而是：

- “如何把现有系统主干，转化成一个开始像 war3 的垂直切片”

### 2.3 当前仍然存在的结构问题

虽然主干已成型，但这些问题依旧存在：

1. `Game.ts` 仍然过大
- 说明模块边界还没有真正稳定。

2. 命令层还只是基础版
- `move / attack / gather / build / train` 已有，但 `stop / hold / attack-move / patrol / rally` 还没建立。

3. 模拟真相仍然部分依赖 Three.js 对象
- 这意味着未来英雄、技能、回放、复杂 AI 还会继续受限。

4. 视觉垂直切片还没开始
- 当前画面不足以证明“这个项目最终会像 war3”。

5. W3X 还主要服务运行时接入，不等于已经有高质量玩法地图
- 现在需要一张“专门为手感和演示服务”的展示地图，而不是继续完全依赖测试地图。

## 3. 之前计划的核心问题

旧版 `PLAN.md` 的问题，不是方向完全错，而是缺少动态调整。

它主要有 4 个偏差：

### 3.1 过早把“完整 ECS/EventBus 大重构”当成主线

旧计划把很多架构理想放在 very early 阶段，但当前项目更适合：

- 先做最小稳定骨架
- 再围绕真实场景逐步抽离
- 而不是先把一整套“看起来正确”的架构铺满

### 3.2 把“系统建设”和“玩家感知建设”割裂了

旧计划里视觉阶段和系统阶段相对分离，但 war3 这种项目不是这样长出来的。

正确节奏是：

- 系统推进一段
- 感知层必须同步推进一段
- 否则项目会越来越能跑，但越来越不像 war3

### 3.3 缺少明确的阶段门禁

旧计划列了很多 Phase，但缺少一句最重要的话：

> 什么时候算这个阶段真的过了，什么时候必须暂停继续加功能。

没有门禁，AI 很容易不断扩功能，看起来进展很多，实际项目方向已经漂了。

### 3.4 缺少“展示切片”思维

war3 这种项目不应该只按系统维度推进，还必须按“可感知切片”推进。

例如：

- 一个真正像 war3 的人族主基地一角
- 一张真正能体现树林/路径/矿区关系的地图
- 一段真正有 war3 选择/命令反馈味道的交互

这些不等于做完整游戏，但它们决定项目有没有未来。

## 4. 新的总控原则

从现在开始，整个项目按下面这些原则推进。

### 4.1 一个阶段只解决一个主矛盾

每个 cycle 可以包含 2-3 个子任务，但必须服务同一个主题。

允许：
- “树木 blocker + 资源目标语义”
- “共享可用资源查询 + 军事情境命令”
- “war3 视觉垂直切片”

不允许：
- 一轮里同时做寻路、快捷键、英雄、UI、技能、特效

### 4.2 系统建设与感知建设交替推进

从现在开始，不允许再连续多个 cycle 只做系统而不做 war3 感知层。

后续开发必须遵循：

- 一个系统主干主题之后
- 必须跟一个玩家可感知提升明显的主题

### 4.3 先做可感知的正确，再做完美的正确

例如：

- 先做一张真正像 war3 的展示地图，再做通用地图编辑器
- 先做正确的 `stop / hold / attack-move` 语义，再做完整命令队列
- 先做人族垂直切片，再谈四种族全量铺开

### 4.4 新功能必须同时回答两个问题

任何新实现都要回答：

1. 它能不能自然扩展到完整 war3？
2. 它会不会让当前版本更像 war3，而不是只是“多了个功能”？

如果两者都答不上来，就不该进主线。

## 5. 产品支柱

后续所有规划围绕 4 根支柱展开。

### 支柱 A：RTS 运行时真相

包括：

- 统一命令层
- 资源节点语义
- 占用 / pathing / blocker 真相
- 单位状态与目标语义
- AI 与玩家共用命令和资源语义

### 支柱 B：war3 感知层

包括：

- 镜头角度与视野范围
- 单位/建筑/地图比例
- 地图空间语法（基地区、树林、路径、矿区、开阔区）
- HUD、命令卡、小地图、选中反馈
- 光照、边界氛围、选中圈、攻击反馈、命令反馈

### 支柱 C：完整交互模型

包括：

- 右键智能命令
- stop / hold / attack-move / patrol / rally
- 选择 / 编队 / 子组 / 快捷键
- 技能目标模式
- 英雄、物品、商店交互

### 支柱 D：浏览器可玩性

包括：

- AI 不每帧决策
- 寻路不每帧重算
- blocker 语义统一
- 限制对象数量和 draw call 爆炸
- 控制每个阶段的性能预算

## 6. 当前阶段重定义

### 当前阶段：Systems Prototype Completed / Vertical Slice Not Started

当前最准确的阶段标签不是旧计划里的“Phase 0 前期”，而是：

> 系统原型期基本成型，视觉垂直切片尚未开始。

这句话很重要，因为它决定了优先级：

- 不是回头再做一轮纯架构理想化重构
- 也不是立刻加英雄和多种族
- 而是：
  - 先把剩余关键语义漏洞补平
  - 再开始做一个真正像 war3 的垂直切片

## 7. 新的版本路线图

### v0.2 - Systems Prototype

定义：
- 运行时骨架已建立
- 可加载地图
- 有资源、建造、训练、战斗、AI、树木与金矿资源语义
- 可以跑完整基础循环
- 但视觉和手感还远远不像 war3

当前状态：
- 基本达到
- 仍有少量命令与共享查询语义需要补平

门禁：
- `npm run build` 稳定通过
- 基础 smoke 可过
- 玩家和 AI 共用主要运行时语义
- 不再出现明显“命令/资源/阻挡真相分裂”

### v0.3 - Vertical Slice 01

定义：
- 做出一张“看起来开始像 war3”的可截图垂直切片
- 重点不是更多功能，而是正确的感知层

必须包含：
- 正确镜头语言
- 正确的地图空间组织
- 正确的单位/建筑/矿区/树林比例
- 可信的 HUD / 命令卡 / 小地图框架
- 基础交互反馈（选中、移动、攻击、采集、建造）

门禁：
- 截图发给 war3 玩家，第一反应接近“这开始像 war3 了”
- 一局微型人族基地循环可玩
- 画面不再像测试板

### v0.5 - Alpha: Human vs AI

定义：
- 人族 1v1 AI 形成完整可玩循环
- 玩家愿意认真打一局

必须包含：
- 人族 T1-T2 核心闭环
- stop / hold / attack-move / rally 基本成立
- AI 有资源节奏和进攻威胁
- 战争迷雾基础可用
- 反馈和 UI 已达到可玩级

门禁：
- 15 分钟内能打完一局
- AI 会正常采集、建造、造兵、进攻
- 玩家不会因为操作和反馈问题中途出戏

### v0.8 - Beta

定义：
- 至少 2 个种族完整可玩
- 英雄与技能系统进入主线
- 可以对外分享与测试

### v1.0 - Release

定义：
- 完整网页版 war3 风格单机产品
- 有明确的内容边界与质量标准

## 8. 新的阶段规划

## Phase A：闭合当前系统主干

目标：
- 把当前还剩的语义缺口补平
- 不再让资源、命令、AI 查询出现明显分裂

包含：
- 共享“最近可用资源节点”查询
- stop / hold / attack-move / rally 基础命令
- AI 和玩家共用命令 / 资源查询语义
- 命令中断与状态一致性修补

说明：
- 这是系统原型收尾阶段
- 不宜拖很久
- 目的不是继续沉迷系统，而是为垂直切片让路

## Phase B：Vertical Slice 01 - Human Base Look & Feel

目标：
- 让一张截图和一段 30 秒操作录像，第一次开始像 war3

包含：
- 相机重调（角度、FOV、目标高度、缩放边界）
- 地图比例与地形空间组织重做
- 基地区、树林、矿区、路径、前压空间重构
- HUD / 选中反馈 / 命令反馈 / 小地图视觉框架
- 光照、边界、氛围、阴影、色温
- 单位和建筑展示代理升级（哪怕仍是代理模型，也要比例和风格先对）

说明：
- 这是当前最重要的阶段
- 它会决定项目未来是继续走“越来越像游戏”还是停留在“越来越复杂的系统原型”

## Phase C：Human Gameplay Alpha

目标：
- 以人族为核心，做出真正能玩一局的 1v1 AI

包含：
- 人族完整经济与生产节奏
- 军事情境命令完整化
- 战争迷雾基础版
- AI 经济与战斗节奏打磨
- 小地图点击跳转、基础快捷键、编队

## Phase D：Hero & Midgame

目标：
- 英雄与中期策略成立，让 war3 的灵魂开始出现

包含：
- 一个完整英雄垂直切片
- 技能目标模式
- 物品栏基础版
- 中立野怪和打野循环

## Phase E：Second Race & Beta

目标：
- 增加第二个种族，验证系统没有绑死在人族语义上

包含：
- 第二种族完整接入
- 更复杂的兵种克制与科技树验证
- AI 扩展到双种族

## 9. 接下来推荐顺序

### Immediate Next 0

修一个短小但必要的系统收尾：
- 共享“最近可用金矿”查询，AI 不再因为最近矿耗尽就发呆

### Immediate Next 1

进入军事命令基础：
- `stop`
- `holdPosition`
- `attackMove`
- 必须和现有命令 / auto-aggro / pathing 状态一致

### Immediate Next 2

立刻切到 Vertical Slice 01，而不是继续纯系统扩展：
- 镜头
- 比例
- 地图空间组织
- HUD 框架
- 选中与命令反馈

### Immediate Next 3

在垂直切片基础上继续打磨：
- 一张手工展示地图
- 一组人族基础表现代理
- 基地一角的真实 war3 感

### Immediate Next 4

再回来补足中层交互：
- rally
- 小地图点击跳转
- 编队与基础快捷键

这个顺序的核心思想是：

> 不再连续做很多个“只有系统开发者能感知到的 cycle”，
> 而是系统闭环和玩家感知提升交替推进。

---

## 10. 明确不该现在做的事

以下内容当前不应进入主线优先级：

- 完整 ECS 大重构
- EventBus 全面铺开
- 四种族并行开发
- 完整英雄系统
- 大规模资产接入
- 地图编辑器
- 回放系统
- 单位碰撞与群组移动的全量优化
- 8 方向 / 分层寻路 / HPA* 等高级寻路升级

原因不是它们不重要，而是：

- 当前最大的产品风险不是这些缺失
- 而是现在的东西看起来和 war3 差太远

## 11. 开发协作协议（给 GLM 的约束）

后续让 GLM 推进时，统一遵循下面的协议：

### 11.1 长周期，不碎片化打断

- 一个 cycle 只做一个主题
- 允许 GLM 连续工作较长时间
- 除非真实阻塞，不要频繁中途汇报

### 11.2 每轮必须回答 5 个问题

1. 做了什么
2. 改了哪些文件
3. 如何验证
4. 剩余风险
5. 下一轮只建议 1 个主题

### 11.3 每轮必须说明“这对完整 war3 的意义”

不能只说：
- 修了什么 bug
- 加了什么功能

还必须说：
- 这项改动怎样服务未来完整网页版 war3 的交互、性能或内容扩展

### 11.4 感知层 cycle 必须附带截图或可视结果描述

只靠代码汇报无法判断“war3 感”是否在提升。

所以凡是视觉 / HUD / 镜头 / 地图 / 交互反馈相关 cycle，必须带：
- 截图
- 或明确的可视变化说明

## 12. 风险清单

### 风险 A：越来越能跑，但越来越不像 war3

这是当前最大的风险。

缓解方式：
- 强制插入 Vertical Slice 阶段
- 不允许连续多个纯系统 cycle

### 风险 B：Game.ts 继续膨胀

缓解方式：
- 每次只抽与当前主题强相关的最小模块
- 不做空抽象

### 风险 C：AI、玩家、命令、资源语义继续分叉

缓解方式：
- 共享查询入口
- 共享命令入口
- 共享资源目标语义

### 风险 D：性能在内容增加后失控

缓解方式：
- 关键系统降频
- 不每帧重寻路/AI/小地图
- 持续把 blocker / query / pathing 收敛到统一语义

## 13. 里程碑定义

### M1：系统原型成立
- build 稳定
- 基础资源 / 建造 / 训练 / 战斗 / AI / blocker / pathing 成立
- 当前基本达到

### M2：第一张像 war3 的截图
- 不是测试板
- 不是纯功能面板
- 一眼能看出 war3 味道
- 当前未达到

### M3：第一局愿意继续打的 1v1
- 玩家会采集、建造、出兵、交战
- AI 有节奏和威胁
- 当前未达到

### M4：人族 Alpha
- 人族 T1-T2 + 基础英雄 + 迷雾 + 军事情境命令
- 当前未达到

### M5：双种族 Beta
- 至少两个种族完整可玩
- 当前未达到

## 14. 当前结论

一句话总结当前项目：

> 这不是“缺一点功能”的问题，
> 而是需要把现有系统原型，重新组织成一个以 war3 感知为导向的完整产品计划。

从现在开始，主线顺序应该是：

1. 收尾当前共享查询与命令语义
2. 立即切入 Vertical Slice 01
3. 用垂直切片验证镜头、比例、地图语法、HUD 和 war3 感
4. 在这个基础上再推进人族 Alpha，而不是继续盲堆系统
