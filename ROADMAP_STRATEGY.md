# War3 RE — 战略级开发路线图 (v2)

> 本文档是对 `PLAN.md` 的上层补充，不涉及具体功能实现细节。  
> 关注的是：**为什么这么做、先做什么、什么时候说够了、风险在哪里。**

---

## 一、产品定位与北极星目标

### 1.1 这是什么项目？

War3 RE 不是"war3模拟器"，也不是"war3克隆品"。  
**它是一个用现代Web技术重现魔兽争霸3核心游戏体验的独立游戏项目。**

核心体验定义（必须实现，否则项目没有意义）：

- 一个玩家能在浏览器里打完一局有挑战性的1v1对战
- 操作手感和决策深度让 war3 老玩家感到"熟悉"
- 整局游戏有开局→中期→决战→胜负的完整弧线

### 1.2 北极星指标

**一个老war3玩家，打完一局后说：**  
> *"这基本上就是war3，而且能在浏览器里玩——太牛了。"*

这句话包含三个判断标准：

1. `基本上就是war3` → 核心机制保真度（采集/建造/战斗/英雄/科技树）
2. `能在浏览器里玩` → 无需安装、性能流畅（≥30fps / 200单位场景）
3. `太牛了` → 技术实现的惊喜感，不是凑合

### 1.3 明确不做什么（护城河）

以下内容永久排除在范围之外，避免无限扩展：

| 不做 | 原因 |
|------|------|
| 多人联机对战 | 延迟同步/状态同步工程量是整个项目的2倍，且需要后端 |
| 完整复刻原版剧情战役 | 剧本+触发器系统极度复杂，且版权风险高 |
| 移动端适配 | 操作模型完全不同，相当于另做一个产品 |
| 自定义地图脚本（Jass/Lua） | 脚本解释器是独立的大型子项目 |
| 完整MPQ资源集成 | 版权问题；使用免费社区资源替代 |

### 1.4 War3 的"乐趣内核"分析

在做优先级排序之前，需要先回答一个根本问题：**war3 为什么好玩？**

war3 的乐趣不是一个单一的系统，而是由几层体验叠加而成：

```
第一层：即时反馈（最底层，缺失则"不好玩"）
├─ 点击→单位立刻响应移动
├─ 右键→命令有清脆的视觉/听觉反馈
├─ 单位打架→看到伤害、看到血条变化、看到死亡
└─ 采集→金子有声音，数字在涨，有"赚到了"的感觉

第二层：策略张力（中层，缺失则"无聊"）
├─ 资源有限→每一笔花费都是取舍（出农民还是出兵？升级还是扩张？）
├─ 信息不完整→迷雾带来紧张感（对面在干什么？）
├─ 时间压力→对方可能随时来打你，你必须有节奏感
└─ 兵种克制→不是无脑造最贵的兵就赢

第三层：英雄故事线（高层，缺失则"缺少灵魂"）
├─ 英雄是"你在战场上的化身"
├─ 英雄升级→越来越强→终极技能改变战局
├─ 英雄死了→你心疼，想报仇
└─ 物品系统→每个英雄都有独特的build路线
```

**开发优先级应该严格按照这个层次顺序来：**

- v0.1~v0.3 必须做到第一层完美（操作响应+即时反馈）
- v0.5 必须做到第二层基本成立（有资源张力+有AI威胁+有迷雾）
- v0.8 必须让第三层可用（至少一个英雄完整可用）
- v1.0 三层全面到位

如果第一层做不好，后面全部白费。一个操作延迟50ms的游戏，策略再深也不好玩。

## 二、版本发布策略

PLAN.md 的 Phase 0~8 是开发任务视角，这里定义面向"可发布状态"的版本节点。  
**每个版本都必须是一个可以给人玩的完整体验，不是半成品。**

```
v0.1 Vertical Slice ──  一个完整场景跑通全部系统
v0.3 Pre-Alpha     ──  第一个可分享的可玩demo
v0.5 Alpha ⭐      ──  完整核心玩法（人族1v1 AI）← 项目成败关键
v0.8 Beta          ──  接近完整的游戏体验
v1.0 Release       ──  公开发布版本
```

### v0.1 Vertical Slice（垂直切片）

**目标：一个完整的"微型游戏"，验证所有核心系统协同工作**  
**受众：自己**  
**预计时间：5~8 天（Cycle 1~2）**

垂直切片的定义非常具体——**不是做完所有系统的50%，而是做完一个场景的100%：**

> 一个农民从城镇大厅出来 → 走到金矿采金（不穿墙）→ 回来交金 → 你建了一个兵营 → 训练了一个步兵 → 步兵遇到敌方步兵 → 开打 → 一方死亡 → 你赢/输 → 结算画面

这一个场景看起来简单，但它要求以下系统全部可用且协同工作：

- ECS架构 + EventBus（架构层）
- 寻路（单位不穿墙到达金矿和敌人）
- 资源系统（按team独立，不是全局变量）
- 建造系统（放下兵营，农民建造，进度条）
- 训练系统（兵营出兵）
- 战斗系统（自动攻击、HP减少、死亡判定）
- 游戏状态机（开始→游戏中→结算）
- 基础UI（资源、选中信息、命令卡）

**为什么要先做垂直切片而不是先重构完所有架构？**

因为重构最大的风险是"重构到一半发现方向错了"。垂直切片强制你在一个真实场景中验证每个系统的接口设计。如果 EventBus 的事件定义不对、如果 ECS 的组件拆分不合理，在这个场景里会立即暴露——代价很低，只需要改一个小场景的代码。

如果先把 Phase 0 全部做完（ECS + EventBus + 6个System + 寻路 + 碰撞），然后才组装，发现接口不匹配，重构成本是灾难级的。

**垂直切片与 Phase 0 的关系：**
```
不是：Phase 0 全部完成 → 开始做游戏
而是：做一个最小场景 → 在场景中驱动每个系统的实现 → Phase 0 自然完成
```

对 GLM 来说，这个场景的所有代码可能在1~2天内就写完。但你需要额外2~3天来：
- 逐模块审查代码是否按设计意图实现
- 跑起来看效果、发现集成问题
- 反复调整接口直到各系统协同顺畅

**质量门禁：**
- 上面那个场景可以完整跑通，无crash
- 单位不穿墙、不重叠
- 建筑放置后正确阻挡寻路
- Game.ts 拆分完成，最大的文件不超过400行
- 200个单位同屏不低于30fps

### v0.3 Pre-Alpha

**目标：第一个值得截图分享的版本**  
**受众：技术圈朋友、war3社区**  
**预计时间：v0.1 后 1~2 周（Cycle 3~5）**

v0.3 在 v0.1 基础上增加三样东西：**视觉、声音、AI雏形**

视觉：
- 模型替代几何体（至少人族基础单位+建筑，具体方案见第三章资产决策）
- 选中单位显示绿色底圈、血条浮在头顶
- 移动命令显示绿色目标指示器
- 攻击时有简单的命中反馈（闪白 or 粒子）

声音：
- 选中单位播放语音（哪怕先用一个通用"嗯？"代替）
- 命令确认音效（"是，长官"）
- 攻击命中音效
- 建造锤击声

AI雏形（不需要完美，但必须有）：
- AI有自己的资源池和基地
- AI会自动训练农民、采集
- AI会建兵营、训练步兵
- AI积累5个步兵后向玩家发起进攻

**为什么 v0.3 就要有AI？**

因为没有AI的游戏是"技术demo"，有AI的游戏才是"游戏"。哪怕AI很笨，但能让你有一个明确的"赢了/输了"的体验，这个版本就有意义。而且AI早期进入可以在整个后续开发中持续作为测试工具——你在调试任何系统时，都可以开一局vs AI来验证。

**质量门禁：**
- 截图发给3个不认识你项目的人，问"这是什么游戏"，至少2人能认出 war3 风格
- 可以vs AI打完一局（哪怕AI很笨）
- 有声音反馈（静音状态下也能玩，但有声音时明显更好）
- Chrome DevTools Performance 面板：单帧耗时 < 33ms

### v0.5 Alpha ⭐（最重要的里程碑——项目的 Go / No-Go 决策点）

**目标：第一个可以认真玩一局的版本**  
**受众：war3 玩家朋友（闭测 5~10 人）**  
**预计时间：v0.3 后 2~3 周（Cycle 6~10）**

v0.5 是一个关键的节点。如果 v0.5 不有趣，应该停下来诊断原因，而不是继续堆功能。

v0.5 的定义：一个war3老玩家在不看任何说明的情况下，能用肌肉记忆操作这个游戏，并且想打第二局。

功能列表：
- 人族 T1~T2 完整可用（农民、步兵、火枪手、骑士）
- 建筑：城镇大厅、兵营、农场、伐木场、铁匠铺
- 科技：至少攻击/护甲升级可用
- AI在Normal难度下有真实威胁（15分钟内出兵进攻）
- 英雄：至少圣骑士可用（神圣之光+专注光环+圣盾术，有升级）
- 战争迷雾运行
- 快捷键体系完整（A攻击、S停止、B建造、编队Ctrl+1~9）
- 胜负判定 + 结算界面
- 小地图点击跳转

**这个版本故意不包含的东西（且不应该因为"顺手做了"而加入）：**
- 第二个种族
- T3单位（狮鹫、坦克等）
- 其余3个英雄
- 昼夜循环
- 粒子特效（简单的闪白反馈足够）
- 地图编辑器

**Go / No-Go 决策：**
```
盲测结果评估：

5人中≥3人说"想再打一局" → GO：继续开发v0.8
5人中2人说"想再打一局"   → 暂停：花2周分析原因，调整后二次盲测
5人中≤1人说"想再打一局"  → STOP：项目需要根本性反思
    可能的原因：
    ├─ 操作手感不对（第一层失败）→ 需要回去打磨操作响应
    ├─ AI太弱没有挑战（第二层失败）→ 需要加强AI
    ├─ 无聊（第二层失败）→ 可能是资源平衡有问题，节奏不对
    └─ 画面太差无法代入（视觉及格线没过）→ 需要投入资产质量
```

**质量门禁：**
- 找5个有war3经历的朋友，在不解释规则的情况下打一局
- 记录：每人操作的前60秒（是否本能地知道怎么玩）
- 记录：第一次完成"采金→建造→出兵→打仗"循环的时间（< 5分钟合格）
- 记录：打完后是否主动说"能再打一局吗？"
- AI 在 Normal 难度下必须在 15分钟内造出攻击部队并发起进攻

### v0.8 Beta

**目标：一个完整的游戏，等待打磨**  
**受众：公开测试，war3社区，社交媒体发布**  
**预计时间：v0.5 后 3~5 周（Cycle 11~18）**

- **两个种族可用（人族 + 兽族）**——注意不是4个
- 每个种族科技树完整（T1~T3）
- 英雄系统完整（每族4英雄，技能可用，物品栏可用）
- 3张预设对战地图
- 音效完整，视觉特效基本覆盖
- AI 3个难度档（简单 / 普通 / 困难）
- 编队、快捷键、双击选中同类等操作细节全部到位
- 昼夜循环

**为什么v0.8先做两个种族？**

虽然 GLM 可以快速生成4个种族的代码，但瓶颈在你这端：

```
每增加一个种族，你需要做的（GLM无法替代）：
├─ 寻找/挑选该种族的模型资产：3~5天
├─ 审查 GLM 生成的种族逻辑代码：2~3天
├─ 集成测试 + 种族特殊机制调试：2~3天
├─ 每多一个种族，对战组合数增加：
│   2族 = 3个组合（HvH, HvO, OvO）
│   3族 = 6个组合
│   4族 = 10个组合  ← 平衡性测试量激增
└─ 总计每种族约 2~3 周你的时间
```

先做人族+兽族到精品级，后两个种族在 v0.3 时根据资产管线实际情况决定是否纳入 v1.0（详见4.6节）。

**质量门禁：**
- 发布到 GitHub/itch.io
- 人族 vs 兽族 AI 可以打完整对战
- 连续玩3局（约90分钟），无崩溃、无内存溢出
- 有至少一个外部用户自发分享了链接

### 总时间估算

```
v0.1 Vertical Slice:   Cycle 1~2    (第1~2周)
v0.3 Pre-Alpha:        Cycle 3~5    (第2~4周)     ← 资产获取是瓶颈
v0.5 Alpha:            Cycle 6~10   (第4~7周)
v0.8 Beta:             Cycle 11~18  (第7~12周)    ← 内容量大，需要反复试玩调优
v1.0 Release:          Cycle 19~22  (第12~15周)

总计：约 22 个 Cycle ≈ 12~15 周 ≈ 3~4 个月
```

这个时间基于以下假设：
- GLM-5.1 承担几乎全部代码编写，产出不是瓶颈
- 瓶颈在你这端：设计决策、审查代码、试玩调参、资产获取
- 每天你能投入 2~4 小时审查/试玩/决策（考虑你有主业）
- 没有重大技术死路

**注意：3~4个月是乐观估算。** 下面两个因素可能让实际时间翻倍：
1. 资产管线卡住（找不到合适的模型，MDX解析失败，GLTF风格不统一）
2. GLM 生成的代码集成质量差（模块间接口不匹配，需要大量返工）

如果到第6周 v0.3 还没出来，就需要停下来重新评估资产方案。

## 三、关键技术决策树

### 3.1 资产管线决策（最高优先级风险）

这是整个项目最大的不确定性。**在 v0.3 之前必须做出且锁定决策**，否则视觉方向会一直摇摆。

```
问题：War3 风格的3D模型从哪来？

选项 A：使用 war3-model.js 库解析真实 MDX 文件
  ├─ 优点：完全保真的 war3 视觉
  ├─ 风险：war3-model.js 对骨骼动画支持可能不完整
  ├─ 风险：法律灰色地带（需要从 war3 安装包提取 MPQ）
  └─ 时间投入：2~4 周验证，失败则沉没

选项 B：HiveWorkshop/Sketchfab 社区的 GLTF/OBJ 免费模型
  ├─ 优点：Three.js 原生支持 GLTF，含骨骼动画，无解析风险
  ├─ 风险：视觉风格不统一，需要花时间挑选和调整
  ├─ 风险：可能找不到所有需要的单位模型
  └─ 时间投入：1~2 周搜索+集成

选项 C：低多边形自制风格
  ├─ 优点：完全自控，无版权风险，可形成独特美学
  ├─ 风险：需要建模技能或外包
  └─ 时间投入：持续性投入，每个模型1~3天
```

**推荐策略：分阶段锁定**

```
v0.1 阶段：继续使用几何体（圆柱+球体），不投入资产
v0.3 开发开始时（Cycle 3）：花3~5天做技术调研
  ├─ 第1~2天：尝试加载3个 GLTF 模型（农民、步兵、城镇大厅）
  ├─ 第3~4天：尝试 war3-model.js 加载1个 MDX 模型
  └─ Cycle 3 结束时做决策：
      ├─ GLTF 成功 + MDX 成功 → 选 MDX（保真度更高）
      ├─ GLTF 成功 + MDX 失败 → 选 GLTF
      ├─ GLTF 失败 + MDX 成功 → 选 MDX
      └─ 都失败 → 继续用几何体，考虑选项 C，v0.3 推迟
```

**无论选哪个，模型层必须有抽象接口：**
```typescript
interface UnitRenderer {
  createUnit(type: string, team: number): THREE.Object3D;
  playAnimation(obj: THREE.Object3D, name: 'idle'|'walk'|'attack'|'death'): void;
  dispose(obj: THREE.Object3D): void;
}
```
这样后期切换方案只改一个实现类。

### 3.2 寻路决策（v0.1 关键）

A* 寻路有已知的性能台阶，在选型阶段就要规划好扩展路径：

```
当前地图规模：  64x64   → 简单 A* 足够
v0.5 目标：    128x128  → 需要 JPS 优化或 Flow Field
v1.0+ 目标：   256x256  → 需要 HPA*（层次寻路）

建议：
- v0.1: 实现标准 A*，格子大小 = 0.5个单位半径
- v0.5: 加入 JPS 优化（A* 基础上改动不大，性能提升 5-10x）
- v0.8: 按需加入 HPA*（区域层+格子层两层寻路）
```

**关键设计决策：寻路频率控制**

不要每帧为每个单位重新寻路。推荐策略：
- 单位收到移动命令时寻路一次
- 每2秒检查一次路径是否被阻挡（建筑新建或树被砍倒）
- 被阻挡时才重新寻路
- 多个单位去同一目标时使用 Flow Field（一次计算，所有单位共享）

寻路抽象接口应从一开始就定义好：
```typescript
interface PathFinder {
  findPath(start: Vec2, end: Vec2, unitRadius: number): Vec2[];
  isWalkable(x: number, y: number): boolean;
  updateBlocker(x: number, y: number, blocked: boolean): void;
}
```

### 3.3 ECS 深度决策

不建议引入第三方 ECS 库（bitecs, @lastolivegames/becsy 等）。  
理由：war3 游戏逻辑高度定制，第三方 ECS 的约束会带来摩擦，而收益（cache性能）在<1000实体规模下不明显。

**推荐：手写轻量级 ECS，只实现必要部分：**
- Entity = number（整数ID）
- Component = 普通 TypeScript class
- System = 有 update(dt) 方法的类，World 统一调度
- 不做 SoA（Structure of Arrays）优化，保持代码可读性

当单位数量超过500导致性能问题时，再评估是否引入 bitecs。

**但更重要的是：不要为了ECS而ECS。**

看一下现在 Game.ts 里的 Unit 接口：
```typescript
interface Unit {
  mesh, type, team, hp, maxHp, speed, moveTarget, isBuilding,
  state, gatherType, carryAmount, gatherTimer, attackTimer, attackTarget,
  armor, attackDamage, attackRange, attackCooldown,
  buildProgress, builder, buildTarget,
  trainingQueue
}
```

这个接口其实已经隐含了组件拆分：
- Transform 组件 → mesh.position
- Health 组件 → hp, maxHp, armor
- Combat 组件 → attackDamage, attackRange, attackCooldown, attackTimer, attackTarget
- Gatherer 组件 → gatherType, carryAmount, gatherTimer
- Builder 组件 → buildProgress, builder, buildTarget
- Trainer 组件 → trainingQueue
- Movement 组件 → speed, moveTarget
- State 组件 → state

拆分的目标不是性能，而是**让每个系统只关心自己的数据**。CombatSystem 不需要知道 trainingQueue 的存在。

### 3.4 性能预算（Frame Budget）

目标帧率 30fps = 每帧 33ms。分配方式：

```
┌──────────────────────────────────────┐
│          33ms 帧预算分配              │
├──────────────────────────────────────┤
│ 渲染（Three.js draw calls）   16ms   │  ← 这个最重，需要监控
│ 游戏逻辑更新                   8ms   │  
│   ├─ 移动 + 碰撞              3ms   │
│   ├─ 战斗 + 死亡              2ms   │
│   ├─ AI 决策                  2ms   │  ← 不在每帧运行，每0.5s一次
│   └─ 其他系统                  1ms   │
│ 寻路                           4ms   │  ← 不在每帧运行，分帧处理
│ UI 更新                        3ms   │
│ 余量                           2ms   │
└──────────────────────────────────────┘
```

**性能红线（任何一项触发立即优化）：**
- 单帧超过 50ms（掉到20fps以下）
- 内存增长超过 10MB/分钟（内存泄漏）
- Draw calls 超过 200（需要合批/实例化）

**性能优化不做提前优化，但必须提前监控。**
从 v0.1 开始，每次 F12 打开 Performance 面板看一眼，心里有数。

## 四、开发模型：人机分工与节奏

### 4.1 真实的团队结构

```
┌──────────────────────────────────────────────────────────┐
│  你（dew）= 制作人 + 美术总监 + QA + 唯一的玩家代表      │
│  ├─ 定义"做什么"和"什么算做好了"                          │
│  ├─ 审查 GLM 输出的代码质量和集成正确性                    │
│  ├─ 搜集和挑选资产（模型/音效/纹理）                       │
│  ├─ 试玩、调参、判断手感                                  │
│  └─ 做所有需要"看到效果"才能决定的事                       │
│                                                          │
│  GLM-5.1 = 全职程序员，7×24，无限产出                      │
│  ├─ 按你的 spec/prompt 生成完整模块代码                    │
│  ├─ 重构、拆分、重写——对它来说成本接近零                    │
│  ├─ 批量生成数据（4个种族的单位定义、科技树、AI策略表）      │
│  └─ 实现算法（寻路、视野、碰撞、伤害公式）                  │
│                                                          │
│  Claude = 架构顾问（偶尔）                                │
│  └─ 设计评审、风险分析、方案比较                            │
└──────────────────────────────────────────────────────────┘
```

### 4.2 瓶颈已经不是代码了

传统的游戏开发项目，瓶颈是"写代码太慢"。你的项目完全不是这样。

```
GLM 可以在几小时内产出的东西：
├─ 完整的 ECS 框架（World + Entity + Component + System）
├─ A* 寻路 + JPS 优化
├─ AI决策树（经济/建造/战斗）
├─ 4个种族全部单位/建筑的数据定义
├─ 英雄技能系统框架 + 16个英雄的技能实现
├─ 战争迷雾 shader
├─ 音效管理器
└─ 几乎所有纯逻辑代码

你必须亲自做、GLM 无法替代的：
├─ 审查：GLM 写的 A* 是不是真的不穿墙？需要跑起来看
├─ 资产：war3 风格的模型从哪找？下载哪个？效果好不好？
├─ 调参：AI 多久出兵才有压迫感？攻击音效是不是太响？
├─ 集成：5个模块各自能跑，拼在一起能不能跑？
├─ 决策：MDX 加载失败了，是继续debug还是切GLTF？
└─ 试玩：这个游戏好不好玩？
```

这意味着：**你的项目进度不取决于 GLM 能多快写代码，而取决于你每天能审查+试玩+决策多少。**

### 4.3 开发节奏：Cycle 而不是 Sprint

传统2周Sprint对这个项目太慢了。推荐使用更短的 **Cycle（循环）**，每个 Cycle 2~4 天：

```
Cycle 结构（2~4天）：

Day 1 - 规划 + 生成
├─ 你写好这个 Cycle 的目标和详细 spec（1~2小时）
├─ GLM 生成代码（它可能几分钟到几小时就写完）
└─ 你快速浏览生成的代码结构，看有没有方向性错误

Day 2 - 集成 + 调试
├─ 把 GLM 生成的代码集成到项目中
├─ 跑起来，发现 bug，让 GLM 修
├─ 反复 review → fix → test 直到功能基本可用
└─ 如果发现架构问题，当天就让 GLM 重写（反正重写对它很便宜）

Day 3 - 试玩 + 打磨（如果需要）
├─ 完整地玩一遍当前版本
├─ 记录：什么感觉不对？什么需要调参数？
├─ 让 GLM 做微调（数值、时机、UI布局）
└─ 确认这个 Cycle 的功能达标

[可选 Day 4] - 如果是资产相关的 Cycle
├─ 搜索/下载模型
├─ 测试加载效果
└─ 反复调整直到视觉满意
```

**纪律：每个 Cycle 结束必须可玩。**  
GLM 生成了一堆代码但游戏跑不起来？这个 Cycle 没有完成。  
宁可功能少一点，也要保证每次都是一个可以运行的完整游戏。

### 4.4 GLM 使用中的核心纪律

**代码产出无限 ≠ 质量自动保证。** 这是 AI 驱动开发最大的陷阱。

```
❌ 危险模式："让 GLM 一口气把 Phase 0~2 全写了"
  → 生成 20 个文件、8000 行代码
  → 你看不过来，随便扫一眼就合进去
  → 3 天后发现 EventBus 和 CombatSystem 之间接口不匹配
  → 推倒重来，比没写还糟

✅ 正确模式：每个 Cycle 只让 GLM 做 1~2 个模块
  → 比如这个 Cycle 只做 "EventBus + MovementSystem"
  → GLM 写完，你跑起来验证：单位移动是否通过事件驱动？
  → 确认没问题，下个 Cycle 再加 CombatSystem
  → 每次增量都建立在已验证的基础上
```

**GLM 重写成本几乎为零——善用这一点：**
- 如果 GLM 写的某个模块你看着不舒服，**直接让它重写**，不要试图人工修补
- 如果两个模块集成后有冲突，让 GLM **用新的接口约定从头生成其中一个**
- 不要对 GLM 的代码有"沉没成本心理"——它5分钟就能重写的东西，不值得你花2小时修

**但你的审查时间有成本——所以要控制每 Cycle 的代码量：**
- 单个 Cycle 新增代码不超过 **1000~1500 行**（你能在2~3小时内审完）
- 如果某个功能需要更多代码，拆成多个 Cycle

### 4.5 Cycle 分配方案

```
Cycle 1~2 (v0.1，约第1周):
  C1: ECS核心 + EventBus + 从Game.ts迁移MovementSystem和CombatSystem
  C2: 寻路(A*) + 碰撞 + TeamResources + 游戏状态机 + 垂直切片验证

Cycle 3~5 (v0.3，约第2~4周):
  C3: AI经济决策 + AI建造 + AI出兵攻击

  C5: 模型替换集成 + 基础音效 + 首次"像war3"的截图

Cycle 6~10 (v0.5，约第4~7周):
  C6: 人族T2单位（火枪手、骑士）+ 更多建筑（伐木场、铁匠铺）
  C7: 英雄系统框架 + 圣骑士（3技能+终极技能）
  C8: 快捷键 + 编队系统 + 战争迷雾
  C9: 科技升级（攻防升级）+ AI增强（使用T2单位、升级）

Cycle 11~18 (v0.8，约第7~12周):

  C13: 物品系统 + 商店
  C14~15: 兽族全部内容（单位+建筑+科技树）← GLM 批量生成数据+逻辑

  C17: 3张对战地图 + AI难度分级 + 音效补全

Cycle 19~22 (v1.0，约第12~15周):
  C19: 平衡性细调（人族 vs 兽族对战数据分析）
  C20: 视觉打磨（粒子特效、昼夜循环、死亡动画）
```

### 4.6 关于种族数量的重新评估

第二版文档建议 v1.0 只做2种族，理由是"每种族要11~14周"。但那个估算基于人类开发者的编码速度。

在 GLM 驱动下重新估算：

```
每个种族的工作量（GLM 承担代码）：
├─ 数据定义（单位/建筑/科技树 JSON）：GLM 半天
├─ 逻辑代码（种族特殊机制）：GLM 1~2天
├─ 4个英雄 × 4个技能：GLM 1~2天
├─ AI 策略适配：GLM 半天
├─ 小计 GLM 工作：3~5天

你需要做的：
├─ 审查代码 + 集成测试：2~3天
├─ 寻找该种族的模型资产：3~5天   ← 这才是真正的瓶颈
├─ 平衡性测试：3~5天             ← 6个对战组合（HvH,HvO,HvU,OvO,OvU,UvU）
├─ 小计你的工作：8~13天 ≈ 2~3周/种族
```

所以：
- **如果模型问题解决了**（选项B的GLTF方案可行），4种族在 v1.0 是可能的。总时间从15周延长到约20周。
- **如果模型问题没解决**（卡在资产获取上），保持2种族 v1.0 的计划。

**决策点：在 v0.3 发布时（约第4周）根据资产管线的实际情况做出最终决定。**

### 4.7 你的精力管理

虽然 GLM 不会累，但你会。新的风险形态：

```
信号：GLM 产出了大量代码，你堆积了 3000+ 行未审查的代码
应对：立即停止让 GLM 生成新东西。花1~2天只做审查和集成。
     未审查的代码是定时炸弹，越积越难处理。

信号：你连续3天都在"修 GLM 写的 bug"而不是推进新功能
应对：说明这个模块的 spec 写得不够好，或者 GLM 对项目上下文理解不足。
     与其继续修，不如重写 spec，让 GLM 用更详细的约束条件重新生成。

信号：你开始觉得"反正 GLM 能写，不如把所有功能都加上"
应对：这是 GLM 驱动开发最危险的范围蔓延形式。
     代码产出无限 ≠ 你的审查/测试/资产获取时间无限。
     每多加一个功能，你就多了审查、集成、测试、调优的工作。
     严格按照第五章的 CORE/POLISH/DEFER 分类执行。

信号：项目跑起来了但你"不想玩"
应对：这比任何技术问题都严重。停下来想想：
     是操作手感的问题？还是游戏节奏的问题？还是视觉不到位？
     找到具体的"哪里不对"，然后集中修那一个点。
```

### 5.2 削减触发条件

当以下任一情况发生时，触发范围缩减评审：

1. 某个 Cycle 的实际时间超出计划2倍
2. 某个技术方案调研超过1周没有结论
3. 同一个 bug 修了3次还复现

**削减决策：**
- DEFER → 立即移出当前版本
- POLISH → 推迟到下一个版本
- CORE → 不能砍，但可以简化（例：英雄技能砍到只做伤害数字，特效后补）

### 5.3 技术债管控

**允许的技术债（有意识地接受）：**
- 硬编码的AI建造顺序（后面会做配置化）
- 不最优但正确的性能（先跑通再优化）
- 暂时跳过单元测试（游戏逻辑的单测收益较低）

**不允许的技术债（一旦发现立即修复）：**
- 超过400行的文件（上帝类复发）
- 跨系统的直接引用（必须通过 EventBus 或接口）
- `any` 类型超过3处（TypeScript 的类型安全是免费的质量保障）
- 未释放的 Three.js 对象（GPU内存泄漏是累积性的灾难）

## 七、GLM-5.1 驱动开发规范

### 7.1 给 GLM 的任务规模控制

GLM 可以一次性生成一个完整系统。但**你不应该让它这么做**，除非你确定自己能在同一天审查完。

```
推荐的单次任务粒度：
├─ 小任务（1个文件，<200行）：接口定义、单个组件、工具函数
├─ 中任务（2~3个文件，<600行）：一个完整 System + 对应的 Component
├─ 大任务（4~6个文件，<1500行）：一个完整子系统（如寻路、或AI决策）
└─ 禁止：一次让 GLM 生成整个 Phase 的全部代码

原因：你审查 1500 行代码需要 2~3 小时。超过这个量，审查质量会断崖式下降。
     而未被认真审查的代码，迟早会在集成阶段炸掉。
```

### 7.2 给 GLM 写 Spec 的模板

GLM 的输出质量直接取决于你的 spec 质量。Spec 越详细，返工越少。

**好的 spec（GLM 一次出稿即可用）：**
```
"实现 CombatSystem，要求：
- 路径：src/ecs/systems/CombatSystem.ts
- 依赖：src/ecs/World.ts, src/ecs/components/Combat.ts, src/ecs/components/Health.ts
- 接口：class CombatSystem implements System { update(dt: number): void }
- 逻辑：
  1. 遍历所有有 Combat + Transform 组件的实体
  2. 在 AGGRO_RANGE 内寻找敌方实体（team不同）
  3. 如果有攻击目标且在 attackRange 内，执行攻击
  4. 伤害公式：实际伤害 = attackDamage × (1 - armor × 0.06)
  5. 目标HP<=0 时，通过 EventBus 发送 'unit:destroyed' 事件
- 常量从 GameData.ts 导入，不要硬编码
- 参考 src/game/GameData.ts 的代码风格
- 文件不超过 150 行"
```

**差的 spec：**
```
"做一个战斗系统"
```

### 7.3 代码审查清单（快速过一遍，2分钟/文件）

- [ ] 文件行数 < 400？
- [ ] 没有 `any` 类型？
- [ ] 没有未使用的 import？
- [ ] Three.js 对象有 dispose 清理？
- [ ] 硬编码数字都用 GameData 常量替代了？
- [ ] 模块间通信走 EventBus，不直接引用其他 System？
- [ ] 新接口和现有代码风格一致？

### 7.4 GLM 输出的常见问题模式

根据大模型代码生成的已知弱点，重点检查：

```
1. "看起来能跑但逻辑错误"
   ├─ 典型：寻路写了A*框架，但启发函数用了曼哈顿距离导致8方向移动时路径不最优
   ├─ 防范：对算法型代码，要求 GLM 同时生成测试用例
   └─ 防范：跑起来看实际效果，不要只看代码

2. "模块内完美，模块间不匹配"
   ├─ 典型：CombatSystem 发送 'unit:died' 事件，但 DeathSystem 监听的是 'unit:destroyed'
   ├─ 防范：在 spec 中明确事件名称，或在一个 GameEvents.ts 中统一定义
   └─ 防范：每新增一个模块就跑一遍完整场景

3. "过度设计"
   ├─ 典型：你要一个简单的AI，GLM 写了行为树+效用AI+GOAP三层架构
   ├─ 防范：在 spec 中明确 "用最简单的方案实现"
   └─ 防范：spec 中给出行数上限

4. "缺少清理逻辑"
   ├─ 典型：创建了 Three.js 的 Mesh 但没有 dispose，GPU 内存泄漏
   ├─ 防范：在 spec 中要求 "必须实现 dispose/cleanup 方法"
   └─ 防范：审查时 Ctrl+F 搜索 "new THREE."，检查每个创建都有对应清理
```
